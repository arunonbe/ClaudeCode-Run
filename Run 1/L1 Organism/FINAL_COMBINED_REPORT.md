## 1. Executive Summary

### What Was Analysed

This report is the definitive synthesis of a complete static analysis of Onbe's 363-repository technology estate, conducted in May 2026. Every repository was analysed by five specialist agent viewpoints (Business Analyst, Data Architect, DevOps/Operations, Enterprise Architect, Solution Architect), producing 1,815 per-repository analysis files. These were synthesised into 75 domain-level documents across 15 business domains, then into 5 master view documents, and finally into this single authoritative report. No repository was excluded. Coverage is 363/363 (100%).

### The Estate at a Glance

Onbe operates three simultaneous platform generations, none of which is capable of independent decommission without the others. Gen-1 (eCount/Citi heritage, pre-2015, approximately 150 repositories) carries an estimated 55–65% of live transaction volume on Java 5–8 runtimes, Apache Axis 1.4 SOAP, a proprietary XML-RPC service bus, and Windows-hosted batch processes reading from a 20-year-old SQL Server OLTP core (ecountcore). Gen-2 (Wirecard/Northlane, 2015–2021, approximately 140 repositories) carries 25–35% of volume on containerised Spring Boot 2.x services running on AWS ECS in what remains an account and DNS namespace bearing a bankrupt company's name. Gen-3 (NexPay/Onbe, 2021-present, approximately 60 repositories) carries 5–15% of volume on Azure Container Apps and AKS, Java 21–25, Spring Boot 3.x–4.x, and PostgreSQL — it is Onbe's strategic investment platform, but it is not production-safe at scale in its current state.

### Finding Severity Distribution

The full-estate analysis produced:

- **29 P0 (Critical — act immediately):** Confirmed code-level findings representing active regulatory violations, unauthenticated access to production payment infrastructure, committed cryptographic key material, and functional bypasses of OFAC and PCI DSS controls.
- **47–74 P1 (High — remediate within 30 days):** Significant security gaps including SQL injection in financial stored procedures, OTP tokens in plaintext logs, non-compliant PAN masking, a confirmed duplicate-disbursement bug in production, and EOL framework CVEs with no patch path.
- **31–100+ P2 (Medium — remediate within 90 days):** Data retention gaps, analytics access controls, PRNG weaknesses, missing IaC coverage, and observability deficiencies.

The Business Analyst, Data Architect, Enterprise Architect, and Solution Architect viewpoints are consistent in their P0 count of 29 discrete findings. The DevOps view identifies an additional tier of P0-equivalent operational findings (JVM debug ports in production, logstash-ship ECS service scaled to zero, the chargeback service requiring a JDK version that has been EOL since 2015). All confirmed P0 findings are enumerated in Section 4.

### The Single Most Important Message

The estate has more than 100 distinct production credential values committed to version-controlled repositories, an unauthenticated platform-wide credential registry (Director) that returns all database passwords to any network-reachable host, and no OFAC/SDN screening on any cross-border wire or Gen-2 ACH disbursement rail. These three conditions in combination mean that any internal actor with repository read access can obtain production database credentials, retrieve all cardholder-encrypted data via the unauthenticated StrongBox vault, and initiate international wire transfers to OFAC-designated entities without triggering any automated control. This must be treated as an active security incident requiring immediate CISO-level escalation. The actions listed in Section 9 (Immediate — This Week) cannot be deferred to sprint planning.

## 2. Estate Overview and Three-Generation Architecture

### Gen-1: eCount / Citi Heritage (pre-2015)

- **Repository count:** approximately 150 repositories (Domains 01, 06, 07, 08, and the core of Domains 02, 04, 05, 10)
- **Production traffic share:** 55–65%
- **Technology stack:** Java 1.5–1.8 compile targets; Spring 1.x–2.5.x XML configuration; Apache Axis 1.4 SOAP (RPC-encoded, WS-I non-compliant; EOL 2006); xml-rpc_LIB proprietary XML-RPC bus as the universal inter-service transport; Struts 1.3.x web tier (EOL 2013; multiple unpatched RCE CVEs); Log4j 1.2.x (CVE-2019-17571, EOL 2015); Tomcat 8.5 (EOL March 2024); Windows Server IIS/Tomcat on-premises deployment with hardcoded `D:\c-base\` filesystem paths; stored-procedure-centric data access across SQL Server (ecountcore, cbaseapp, jobsvc, ordersvc); Sterling NDM/Connect:Direct for file exchange; VBScript operations via jIntegra J2COM bridge.
- **Key services:** director-svc_SVC (credential and service registry for the entire Gen-1/Gen-2 estate), ecount-core_SVC (card lifecycle), emboss-extract_LIB (card bureau egress), auto-card-batch_LIB (batch card issuance), jobservice_SVC / job-scheduler_SVC (batch disbursement orchestration), strongbox-xmlrpc_SVC (cryptographic vault), xsecurity_SVC (operator identity), xsso_SVC (SSO), clientzone_WAPP, csa_WAPP, notification-framework_SVC.
- **Dominant risks:** PAN transmitted in plaintext to card bureaus via emboss-extract_LIB; CVV potentially stored in ecountcore fdr_card_account_detail post-authorisation; unauthenticated Director credential registry exposing all production database passwords; unauthenticated XML-RPC dispatch; Java 5/6 EOL runtimes on production payment paths; MD5 password hashing; 3DES with fixed IV in SSO; no automated testing on any Gen-1 service.

### Gen-2: Wirecard / Northlane (2015–2021)

- **Repository count:** approximately 140 repositories (Domains 02, 03, 13, and supporting infrastructure in Domains 12, 05)
- **Production traffic share:** 25–35%
- **Technology stack:** Spring Boot 1.5–2.5; Java 8–21; AWS ECS Fargate containerisation; Docker; Spring Cloud Config Server; ActiveMQ and TIBCO JMS; OAuth 2.0 JWT service-to-service authentication (where wired); SFTP/PGP bank file exchange; Oracle 11g/12c (Wirecard issuing platform); AWS Route 53, ALB, ECR, SQS managed via Terraform.
- **Key dependencies still in use:** nam.wirecard.sys Active Directory domain (Wirecard AG entered insolvency 2020; DNS governance post-acquisition is unresolved); wirecard.com DNS entries in production configuration; director-client_LIB (Gen-2 services still depend on Gen-1 Director for credentials at startup); xml-rpc-clients_LIB (Gen-2 services still call Gen-1 eCountCore via XML-RPC); logstash-ship ECS service (currently scaled to zero — the PCI DSS Req 10 log archive is inactive); `wc-root-state` S3 bucket (Wirecard-era AWS account with no QA/production account separation).
- **Dominant risks:** TLS certificate validation disabled JVM-wide in cbts-client_LIB for all Cambridge FX traffic; PGP and RSA private keys committed to repositories; no OFAC screening in any cross-border disbursement path; Cambridge beta endpoint hardcoded as default in cambridge-service_LIB; Spring Boot 1.5.x EOL; BouncyCastle 1.48 (2012); Quartz 1.6.x and Oracle dependencies in the Wirecard issuing platform.

### Gen-3: NexPay / Onbe (2021+)

- **Repository count:** approximately 60 repositories (Domains 03, 05, 09, 11, and IaC in Domain 12)
- **Production traffic share:** 5–15%, primary investment
- **Technology stack:** Java 21–25; Spring Boot 3.x–4.x; Azure Container Apps (ACA) and AKS; Azure PostgreSQL Flexible Server; Azure Key Vault with Managed Identity; Dapr sidecar for secrets and pub/sub; OpenAPI-first contracts; Flyway schema migrations; OpenTelemetry with Dynatrace OTLP; CycloneDX SBOM; Microsoft Entra External ID for cardholder authentication.
- **Production readiness gaps:** Saga compensation logic is a no-op stub in both orchestrators (nexpay-order-orchestrator, nexpay-recipientorchestrator-svc) — card reversals do not execute on saga failure. recipient-screening-api uses `anyRequest().permitAll()` — OFAC enforcement gate is fully unauthenticated and injectable. om-payment-api `JwtSecurityValidator.java:57` returns `true` unconditionally — all payment operations (PAN retrieval, CVV retrieval, addFunds, withdraw) are exposed to any caller. nexpay-ivr-bff returns hardcoded SSN `987654321` and a static card number via an external APIM-routed endpoint. nexpay-parent POM is `0.2.8-SNAPSHOT` — non-reproducible builds across all Gen-3 services. No production Terraform definitions exist in nexpay-iac — production infrastructure is provisioned outside IaC governance. Azure App Configuration public network access is enabled; Key Vault purge protection is disabled.

### The Five Architectural Blockers Preventing Gen-1 Decommission

**1. Director (director-svc_SVC / director-client_LIB):** 90–110 repositories retrieve all production database passwords, API keys, and SMTP credentials from Director at startup via an unauthenticated HTTP endpoint (`/dispatch.asp`). Director is simultaneously the primary platform-wide SPOF — its failure cascades to all dependent services — and an unauthenticated platform-wide credential exposure. No service can achieve cloud-native autonomous deployment while Director remains the credential broker.

**2. xplatform / cBase XML-RPC bus (xml-rpc_LIB):** The proprietary `application/x-mapxml` wire format is embedded as a JAR in every Gen-1 and Gen-2 service WAR. It is the transport for all inter-service calls. It has no authentication, no operation allowlist, and no equivalent in any cloud-native service mesh or API gateway. A single vulnerability in xml-rpc_LIB simultaneously compromises all consuming services. Replacement requires either a simultaneous coordinated fleet-wide migration or a protocol bridge layer.

**3. JobService distributed scheduler (jobservice_SVC, job-scheduler_SVC):** Every dollar disbursed via batch flows through JobManager and JobAgent. No Gen-3 equivalent exists. scheduler_WAPP (the scheduling interface) has an unauthenticated Java-deserialization RCE surface and is a Spring HTTP Invoker migration blocker (Spring HTTP Invoker was removed in Spring 6). An outage halts all batch client disbursements across all programs with no automated failover.

**4. EcountCore stored-procedure DAL:** The DS_DB_ecountcore SQL Server database has been in continuous operation for approximately 20 years. Its 300+ stored procedures contain embedded fee calculations, velocity checks, escheatment rules, and NACHA timing logic that has never been extracted into service-layer code. It is the system of record for all prepaid card accounts; every analytical, reporting, and card-lifecycle capability is downstream of this database. Zero-downtime decomposition is a multi-year programme requiring dedicated programme-level governance.

**5. SQL Server on-premises estate (50+ databases):** Beyond ecountcore, the platform operates cbaseapp, jobsvc, ordersvc, riskdb, strongbox, notificationsvc, repositorysvc, and 15 per-server instance databases. Gen-3 services connect directly to these on-premises databases via VNet peering, creating a hybrid CDE network topology. The entire analytical layer's ETL credentials are bound via DPAPI to a single Windows user account (NAM\nick.doan) on a single server (P-NA-DB11). Loss of that account permanently destroys all warehouse ETL credential access with no recovery path.

## 8. DevOps and Operational Risk Summary

### CI/CD Maturity by Domain

The estate carries more than 100 distinct production credential values committed to version-controlled repositories. The highest-severity instances are enumerated in Section 4. ## 10. Analysis Methodology and Coverage

### Analysis Approach

All 363 repositories in the Onbe GitLab estate were processed through a four-phase multi-specialist analysis pipeline:

**Phase 1 — Per-Repository Analysis (363 repos × 5 views = 1,815 files).** Each repository was analysed independently by five specialist agents:
- Business Analyst: business capability, regulatory impact, cross-domain flows, operational risk
- Data Architect: data stores, sensitive data classification, encryption posture, key management, data lineage
- DevOps/Operations: CI/CD maturity (5-level scoring), secrets management, deployment topology, observability gaps
- Enterprise Architect: integration patterns, architectural dependencies, migration blockers, SPOF identification, blast radius
- Solution Architect: code-level security findings with file:line precision, CVE inventory, injection vulnerabilities, authentication failures, technical debt estimation

**Phase 2 — Domain Synthesis (15 domains × 5 views = 75 files).** All per-repository findings within each of the 15 business domains were synthesised into domain-level master documents per viewpoint, cross-referencing findings across repositories and identifying domain-level patterns.

**Phase 3 — Master View Synthesis (5 master files).** All 75 domain synthesis documents were synthesised into 5 estate-wide master view documents, establishing cross-domain patterns, ranking findings by estate-wide impact, and producing integrated recommendations per viewpoint.

**Phase 4 — This report.** Final synthesis of all five master views into a single authoritative document for board and CISO-level use.

**Coverage: 363/363 repositories (100%). All repositories have complete 5-file per-repo analysis sets. All phases used complete 363-repository data — there are no coverage gaps.**

### Confidence Notes and Limitations

**Partial clone repositories:** Two repositories have partial clone data due to Windows MAX_PATH constraints — `exemplar-cross-border-transfer-service_WAPP` and `wirecard_nam-bank-agent_LIB`. Analysis is based on available files; findings are grounded in available source; conclusions may be incomplete relative to full repository content.

**Confirmed empty repositories:** RecipientApp, stip-generated, stip-models, OP_Mobile_TESTING_PT, oneplatform-azureblobtags-function, and GP-RSM-Customization contain only git metadata with no source code. These are noted as governance gaps in the applicable domain analyses (Domains 03, 09, 11, 13, 14). Security and compliance review cannot be completed for components with no source. **Analysis type:** Static analysis only. This report does not include runtime traffic capture, active penetration testing, database instance access (database schemas and content are inferred from stored procedures, RowMappers, JPA entity definitions, and Flyway migration files in application code), infrastructure configuration verification, or network segmentation testing beyond what is derivable from application-layer configuration files and IaC code. Runtime conditions may vary from static analysis conclusions.

**The ecountcore CVV finding** (P0 Finding No. 8) is characterised as "potentially stored post-authorisation" because the definitive determination requires a direct SQL query against the production ecountcore database, which is beyond the scope of static analysis. The finding must be treated as confirmed for risk management, remediation prioritisation, and QSA notification planning purposes until a production database inspection confirms or rules out live CVV storage.

**Limitation on penetration testing:** This assessment does not include active penetration testing. The findings documented represent code-level confirmed vulnerabilities. The actual attack surface and exploitability of each finding in the production environment may differ from static analysis conclusions, and may be higher (additional runtime configurations not captured in code) or lower (network controls not visible in source). A dedicated penetration testing engagement is recommended within 90 days as a companion to this assessment, targeting the P0 unauthenticated endpoints documented in Section 4.
