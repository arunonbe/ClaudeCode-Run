## Architecture Position Assessment

### WIRED in the Onbe Product Portfolio Context

WIRED is a **Gen-2 proprietary reporting product** — it was built as a temporary client-facing capability while Onbe/Northlane developed its Gen-3 reporting infrastructure. Key questions for Onbe's enterprise architecture:

1. **Is WIRED still in active client use?** The CCP Oracle DWH source was shut down in July 2020. If cache data has not been refreshed since then, client reports would contain stale data. Either:
   - WIRED has been decommissioned (and clients migrated to Gen-3 reporting)
   - An alternative data feed for cache tables exists (not visible in these repos)
   - Reports are being delivered with stale 2020 data (a serious accuracy risk)

2. **What replaces WIRED in Gen-3?** The Onbe Gen-3 architecture (Azure-native, microservices) would replace WIRED with an API-driven report delivery system. The `oneplatform-rest_API` and related repos in the broader Onbe portfolio may represent this replacement.

3. **Sunrise Banks dependency**: Some WIRED reports may serve Sunrise Banks (the issuing bank) in addition to corporate clients. Bank reporting obligations may have different decommission timelines than client reporting.

## Architectural Assessment

### Strengths
1. **Clean separation of generation and delivery**: Report generation packages (`rpt_*.dtsx`) are separate from delivery packages (`send_wep.dtsx`, `send_client_sftp.dtsx`), allowing delivery method changes without modifying report logic.
2. **Subscription-driven architecture**: The `report_requests` table drives execution — adding a new client subscription does not require package modification.
3. **Execution logging**: `report_requests_log` provides full audit trail of deliveries.
4. **Small focused delivery packages**: `send_client_sftp.dtsx` (1,388 bytes) and `send_wep.dtsx` (7,307 bytes) are appropriately small and focused.

### Weaknesses
1. **`call_report_packages.dtsx` at 59,952 bytes**: This package is likely a large CASE/switch-like control flow. Adding a new report type requires modifying this package. It is a hotspot for change and a maintenance risk.
2. **No retry/resumption logic**: Failed deliveries do not retry until the next scheduled run (next day for daily schedules).
3. **Rendering engine dependency**: If reports are rendered via SSRS/Crystal, the SSIS packages likely call the rendering service (RS Web Service or Crystal runtime). This dependency is not visible in the `.conmgr` files but would be in the package XML.
4. **No delivery confirmation loop**: WEP portal delivery is one-way — no confirmation that the client picked up the report.

## Migration Complexity and Blockers

### Complexity: HIGH

1. **Five distinct report type packages**: Each `rpt_*.dtsx` package likely contains complex data transformation logic tailored to its report type. Migrating to a modern reporting platform (Power BI Paginated Reports, SSRS on Azure, Telerik Reports) requires reimplementing each report separately.

2. **`call_report_packages.dtsx` complexity**: At 60KB, this orchestration package likely contains extensive conditional logic. Full XML analysis is required to enumerate all report routing paths before migration.

3. **WEP portal migration**: `sftp.amer1.wirecard.com` is a Wirecard-branded client portal. Client-facing SFTP endpoints are visible to clients — any endpoint change requires client communication and potentially client-side configuration changes.

4. **Client SFTP endpoint inventory**: Client SFTP delivery targets are stored in `report_requests.DeliverySpecification` — a database field, not source-controlled. The full inventory of active client SFTP endpoints is not visible in this repo.

5. **Subscription data migration**: `report_requests` and `report_requests_log` tables contain active client configuration and historical records that must be migrated to any replacement system.

### Blockers
- Oracle DWH / cache data freshness (if stale, migration must include data source replacement)
- WEP portal migration requires client coordination
- Client-specific SFTP endpoint inventory must be established before migration
- Report rendering engine replacement (SSRS/Crystal → modern equivalent) is a parallel workstream
- Contractual obligations to clients define migration timeline constraints
