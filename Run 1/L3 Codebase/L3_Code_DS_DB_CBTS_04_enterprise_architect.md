# DS_DB_CBTS — Enterprise Architect View

## 1. Platform Generation Assessment

**Generation: Gen-3 (Onbe/Modern Microservice)**

Evidence for Gen-3 classification:

- **Liquibase schema management**: Gen-1/Gen-2 systems use manual DBA scripts or SSDT; Liquibase is a modern, developer-owned migration approach.
- **Spring Batch**: The presence of Spring Batch infrastructure tables confirms a Java microservice architecture — not a SQL Server Agent/SSIS-based job infrastructure.
- **Clean normalised schema**: 22 tables, no God-tables, clear entity relationships with proper FK constraints — consistent with purpose-built domain microservice design.
- **No stored procedures**: Zero stored procedures — all logic in the Java application layer. This is the opposite of Gen-1 cbaseapp (781 procedures) and indicates a clean architecture boundary.
- **UUID-based identifiers**: `VARCHAR(36)` primary keys suggest UUID/GUID identifiers generated by the application, not SQL Server IDENTITY integers — a modern pattern.
- **Audit columns**: `INSERTED_AT`, `INSERTED_BY`, `UPDATED_AT`, `UPDATED_BY` on core tables — consistent with a modern audit-by-convention approach.
- **Version column**: `VERSION BIGINT` on core tables suggests optimistic locking (Hibernate JPA pattern), confirming a Java ORM layer.

## 2. Role in the Onbe Payments Architecture

CBTS is an **isolated domain microservice** for international money movement. It represents a bridge between Onbe's US prepaid card ecosystem and international payment corridors.

```
┌──────────────────────────────────────────────────────────────────┐
│                      ONBE PAYMENTS PLATFORM                      │
│                                                                  │
│  ┌──────────────┐        ┌──────────────────┐                   │
│  │  cbaseapp    │        │  CBTS (this DB)  │                   │
│  │  (cardholder │───────►│  Spring Batch    │                   │
│  │  identity)   │        │  microservice    │                   │
│  └──────────────┘        └────────┬─────────┘                   │
│                                   │                              │
│                                   ▼                              │
│                          ┌──────────────────┐                   │
│                          │  Gateway Provider │                   │
│                          │  (external FX /  │                   │
│                          │   wire transfer) │                   │
│                          └──────────────────┘                   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  dbadmin / database_maintenance (infrastructure)         │   │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

### Upstream
- **Onbe cardholder portal / API**: The CBTS API receives transfer initiation requests from cardholders. The `REMITTER.ACCOUNT_IDENTIFIER` ties back to the cbaseapp/ecountcore cardholder account.
- **Rate quoting engine**: The `RATE` table receives FX quotes from the gateway.

### Downstream
- **External gateway**: `GATEWAY` column on multiple tables references an external payment gateway (likely a specialist cross-border provider such as Earthport, Currencycloud, or similar). Gateway identifiers are stored on `TRANSFER`, `BENEFICIARY`, `RATE`, and `REMITTER`.
- **Reconciliation**: `RECON_FILE` data feeds downstream reconciliation and finance processes.
- **OFAC/compliance screening**: `BENEFICIARY_REGULATORY_RULE` outputs feed compliance checks before transfer execution.

## 3. Integration Patterns

| Integration | Mechanism | Notes |
|---|---|---|
| cbaseapp / ecountcore | API call (REMITTER.ACCOUNT_IDENTIFIER) | CBTS references the Onbe account but does not store PAN |
| External FX gateway | REST API (inferred from GATEWAY_* columns) | All gateway references are opaque IDs stored as VARCHAR |
| Compliance/OFAC screening | BENEFICIARY_REGULATORY_RULE table | Rule outcomes stored per beneficiary |
| Spring Batch scheduling | Spring Batch metadata tables | Self-contained job scheduling and restart |
| Reconciliation | RECON_FILE table + downstream process | File-based T+1 reconciliation |

---

## 4. Architectural Strengths

1. **Domain isolation**: CBTS is cleanly separated from Gen-1 databases. No linked-server queries to ecountcore or cbaseapp.
2. **Schema migration ownership**: Liquibase migrations are owned by the development team — appropriate for a microservice.
3. **Restart resilience**: Spring Batch provides built-in job restart capability after failures, reducing operational blast radius.
4. **Audit by design**: `VERSION`, `INSERTED_AT/BY`, `UPDATED_AT/BY` are present on all core entities.

---

## 5. Architectural Concerns

1. **Single gateway dependency**: If the external gateway provider changes (vendor swap, API version change), all `GATEWAY_*` foreign keys and status values need updating — a potentially high-impact migration.
2. **No encryption at column level**: Foreign bank account data is stored in plaintext — a significant gap in a Gen-3 service that should incorporate a secrets/token vault from the start.
3. **History tables without archival strategy**: `TRANSFER_HISTORY` and `RATE_HISTORY` are append-only with no retention policy, which will cause unbounded growth.
4. **Spring Batch context serialisation**: `SERIALIZED_CONTEXT NVARCHAR(MAX)` columns may contain sensitive data serialised as Java objects. These are not scanned or classified by the schema alone.
