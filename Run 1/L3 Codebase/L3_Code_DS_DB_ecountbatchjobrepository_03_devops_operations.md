## Deployment Approach

The SSDT project model supports two deployment approaches:

1. **Snapshot/DACPAC deployment**: The full schema is compared to the target database and a delta script is generated and applied. This is the expected approach given `DeployToDatabase=True`.
2. **Manual script execution**: Individual `CREATE TABLE` scripts from `dbo/Tables/` are run manually if DACPAC deployment is not used.

Since this database schema is very stable (Spring Batch schema rarely changes between minor versions), deployments are expected to be infrequent — primarily when upgrading the Spring Batch version used by the Java services.

## Backup and Recovery

### Operational Criticality
The `EcountBatchJobRepository` is critical for batch job restartability. If this database is unavailable:
- All Spring Batch jobs will fail to start (Spring Batch checks JobRepository availability on startup)
- Failed jobs cannot be restarted from last checkpoint
- Batch operations across all services (`ECORESVC`, `ECAPSVC`, `SCHSVC`, etc.) are blocked

### Recovery Considerations
- Standard SQL Server backup policies should apply (full nightly, differential every few hours, transaction log every 15-30 minutes for RPO)
- The schema is fully reproducible from this repository in minutes
- The operational data (job history) has limited recovery value beyond audit/investigation use cases
- Point-in-time recovery of job history is only needed for compliance audit investigations

## Monitoring

No monitoring is defined in this repository. Monitoring of the Spring Batch `JobRepository` should include:

1. **Job failure alerting**: Query `BATCH_JOB_EXECUTION` for `STATUS = 'FAILED'` not followed by a successful restart — alert operations team
2. **Long-running jobs**: Alert when `START_TIME` is more than X hours old without `END_TIME`
3. **Abandoned jobs**: Detect `STATUS = 'STARTED'` with `LAST_UPDATED` > 2 hours ago
4. **Table growth monitoring**: Alert when table row counts exceed operational thresholds (e.g., `BATCH_STEP_EXECUTION` > 10M rows without purge)

## Operational Risks

| Risk | Severity | Detail |
|---|---|---|
| No automated deployment pipeline | Medium | Manual DACPAC deployment by DBA; risk of schema drift between environments |
| No table purge mechanism | Medium | BATCH_JOB_EXECUTION and BATCH_STEP_EXECUTION grow unbounded; eventual performance degradation |
| 30+ user accounts with direct DB access | High | Broad access surface; emergency accounts (`emer_*`) with db_datawriter should be audited and time-bounded |
| SQL Server 2012 target | High | SQL Server 2012 is end-of-life (July 2022); upgrade path to SQL Server 2019/2022 required |
| No HA/DR configuration defined | Medium | If the SQL Server instance hosting this DB is unavailable, all Spring Batch jobs stop |
| db_accessadmin and db_securityadmin granted to security accounts | Medium | `NAM\ISA_SQL_SECADMIN`, `ifs_infosec`, `ifs_gidadb`, `scpardb`, `NAM\PROD_CPP_APAC` hold `db_accessadmin` and `db_securityadmin` — these roles can modify database access permissions, which is elevated privilege requiring strict control |

## Role Membership Analysis (`RoleMemberships.sql`)

The `RoleMemberships.sql` file reveals that:
- `NAM\PROD_CPP_APAC`, `NAM\ISA_SQL_SECADMIN`, `ifs_infosec`, `ifs_gidadb`, `scpardb` are granted `db_accessadmin` and `db_securityadmin` — they can add/remove database users and modify role memberships. This is very elevated privilege for what should be a simple batch metadata database.
- `NAM\PROD`, `NAM\UAT`, `NAM\PROD_CPP`, `NAM\PROD_ITOPS` and several `emer_*` accounts are `db_datareader`.
- Several `emer_*` accounts are both `db_datareader` and `db_datawriter` — emergency access logins with write capability should be reviewed and tied to a break-glass access request process.
