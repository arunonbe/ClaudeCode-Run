# account-management-api_API — Business Analyst View

## Business Purpose

The Account Management API is a B2B SOAP web service that exposes prepaid card and disbursement account lifecycle operations to partner clients. It acts as the primary programmatic interface through which Onbe clients create, fund, and manage prepaid accounts and card instruments. The service is positioned behind API Management (published to external APIM at `https://api-qa.onbe.io/accountmanagementapiws/services/AccountManagementApiWebServices`) and serves as a gateway to core processing platforms (ECount/xplatform Order Service, cbase).

The top-level artifact is named `AccountManagementAPI`, version `3.1.8`, parent `com.parents:prepaid-parent:6.0.13`.

## Business Capabilities

The following 18 discrete operations are exposed through `AccountManagementApiWebService` (interface in `accountmanagementapi-ws/src/main/java/.../ws/AccountManagementApiWebService.java`):

| Operation | Description |
|---|---|
| `createAccount` | Create a new prepaid account (optionally with card issuance, load, and card link) |
| `updateAccountStatus` | Change account status (ACTIVE, CLOSED, Activate, etc.) |
| `updateRegistration` | Update cardholder personal/address registration data |
| `updateProvisionStatus` | Update provision/card issuance status |
| `addFunds` | Load funds to an existing account |
| `linkCard` | Associate a card package with an account |
| `assignPackage` | Assign a pre-existing card package to an account |
| `createPackage` | Create a new card package (express or default delivery codes `069`/`000`) |
| `createBulkOrder` | Submit bulk card order |
| `setPin` | Set PIN on a card (limit: 1 PIN set per 24 hours) |
| `withdraw` | Withdraw funds via ACH, ACH Void, Check, or Check Void |
| `activationStatusInquiry` | Query card activation status |
| `activateCard` | Activate a card |
| `cardInquiry` | Query card details (encrypted or JWE format available) |
| `cvvInquiry` | Return CVV value (encrypted with AES-CBC/PKCS5Padding, program shared secret) |
| `getBalance` | Retrieve available balance via Banker service |
| `getRequestStatus` | Poll request processing status |
| `stopPayment` | Stop payment on an ACH or check |

Additional internal operations handled through `ProvisionServiceApiWebService` exist for provision flows.

## Business Entities

- **Program**: The highest-level business grouping. All operations are scoped to a `program_id`. Parsed as a composite ID (product / brand / affiliate) by `ProgramIdUtils`.
- **Promotion / Promotion ID**: A sub-unit within a program that governs disbursement rules. Validated via `PromotionBelongToProgramSP` stored procedure (`dbo.check_promotion_belong_to_program`).
- **Account / Partner User ID**: The cardholder account, identifiable by `accountNumber` (ecount ID) or `partner_user_id` (partner-side alias). Both cannot be passed simultaneously (enforced in `AccountManagementApiServiceImpl`).
- **Card / Card Package**: A physical or virtual card artifact. Card packages have expiry and single-use constraints (`CARD_PACKAGE_EXPIRED`, `CARD_ALREADY_USED`).
- **Registration**: Cardholder PII — name, address, phone, email, date of birth, SSN. Defined in `RegistrationInput`.
- **Load**: Fund transfer descriptor. Contains amount, claimable flag, notification indicator. Defined in `LoadInput`.
- **Transaction**: Idempotency anchor identified by `transaction_id`. Duplicate detection relies on Order Service.
- **Claim Code**: Generated by the Order Service AddFundsActionResult when a promotion is "claimable" (`claimable=1` flag on `LoadInput`, or inferred from `app_profile_global_label` DB query via `CheckClaimableFlagQuery`).
- **End Client Relationship**: A newer entity (`EndClientValidationRequest`, `EndClientRelationshipValidationResponse`) representing a KYC/client relationship validated via an external OAuth-protected microservice.

## Business Rules & Validations

Evidence sourced from `AccountManagementApiServiceImpl`, `BusinessValidationException.BusinessValidationType`, and individual service classes:

1. **Account/Partner User ID mutual exclusivity**: Exactly one of `accountNumber` or `partner_user_id` must be supplied; both or neither throws `MISSING_PARTNER_USER_ID_OR_ACCOUNT_NUMBER` or `BOTH_PARTNER_USER_ID_AND_ACCOUNT_NUMBER`.
2. **Account–Program binding**: `accountNumber` must start with `programId` (`isAccountBelongToProgram()` in `AccountManagementApiServiceImpl` line 571).
3. **Promotion–Program binding**: Promotion ID validated via `PromotionBelongToProgramSP` SP; mismatch throws `PROMOTION_ID_PROGRAM_ID_MISMATCH`.
4. **PIN rate limit**: Only 1 PIN set per 24-hour period (`OVER_PIN_SET_LIMIT`).
5. **Claimable flag logic**: Multi-layered — respects explicit `claimable=1` on `LoadInput` and falls back to DB label query (`CheckClaimableFlagQuery`, column `is_present` from `app_profile_global_label`/`app_profile_promotion_label`).
6. **Security-per-method access control**: Each operation is independently authorized via `SecurityValidator.authorize()` against a named Domain (API=`AccountManagement`, Method=`createAccount`, `addFunds`, `cvvInquiry`, etc.). Feature flags control sensitive data return (`Return-Card-Number`, `Return-VISA-JWE`, `Return-Encrypted-Card`, `Return-CVV`).
7. **Withdraw void rules**: CheckVoid requires exactly one of `original_partner_withdraw_id` or `check_number` (not both, not neither). ACHVoid and CheckVoid validate original transaction existence.
8. **Amount bounds**: `AMOUNT_LESS_THAN_MIN`, `AMOUNT_MORE_THAN_MAX` enforced by Order Service and surfaced through exception handling in `handleOrderExceptions()`.
9. **Address validation**: City-State-Zip consistency validated by downstream Order Service (`INVALID_ADDRESS`).
10. **Account status enumeration**: Valid statuses are `CLOSED|ACTIVE|Activate|activate|ACTIVATE` (configured in `accountmanagementapi.yaml`, line 46).

## Business Flows

### Create Account (with load)
1. Client calls `createAccount` with `CreateAccountRequest` (containing `RegistrationInput`, optional `CardInput`, `LoadInput`, `LinkInput`).
2. `CreateAccountService.execute()` calls `createAccountHelper.prepareCreateAccountRequest()`.
3. `AccountManagementApiServiceImpl.process()` validates security, sends to `SynchronousOrderProcessor.processSweepRequest()`.
4. Order Service returns `ProcessSweepResponse` with `AddFundsActionResult` and `RegisterUserActionResult`.
5. Claim code extracted from result; VirtualExpressURL constructed if non-claimable.
6. Recipient screening performed asynchronously if `RegisterUserActionResult` contains member ID.
7. Card details returned (plain, AES-encrypted, or Visa JWE format) based on security feature flags.

### Add Funds (claimable path)
1. `AddFundsService` receives `AddFundsInput` with `LoadInput.claimable=1`.
2. DB checked for claimable_choice label (`CheckClaimableFlagQuery`).
3. Order Service processes and returns `AddFundsActionResult.claimCode`.
4. If `notificationFlagError` and claim code present, response code is `EMAIL_NOTIFICATION_FAILED` (success with warning).

### CVV Inquiry
1. `CvvInquiryService` authorizes `Return-CVV` feature flag.
2. Retrieves shared secret from `AppProgramSharedSecretProfile` via `ProfileHelper`.
3. Fetches `CreditCard` via `AccountHelper.getCardInfo()`.
4. Encrypts CVV with `CardEncryptionHelper.encrypt()` (AES-CBC, SHA-256 key derivation).
5. Returns encrypted CVV string in `CvvInquiryResponse`.

### Withdraw (ACH)
1. `WithdrawService` resolves `UserMapping`, creates ACH bank device via `AccountHelper.createACH()`.
2. Constructs `WithdrawAction` with withdraw type, amount, express flag, partner withdraw ID.
3. Sends to Order Service; handles duplicate bank (`DUPLICATE_BANK_INFO`, error code 14022).

## Compliance & Regulatory Concerns

- **PCI DSS**: The service handles SAD. CVV is retrieved from core systems (`CreditCard.getCvCode()`) and immediately AES-encrypted before transmission. Card numbers returned only under `Return-Card-Number` feature gate, optionally Visa JWE-wrapped (`JWEHelper`) or AES-encrypted. `CvvInquiryOutput.cvv` holds an encrypted CVV — the field name implies SAD exposure if the authorization gate fails.
- **SSN / Date of Birth handling**: `RegistrationInput` accepts `ssn` and `date_of_birth` fields. These are wrapped in `SecureUserProfile` and stored as `ActionSecureMemo.secureValues` (XML blob) before being sent to Order Service. Exposure during logging is a risk (see `getRequest()` method in `AccountManagementApiServiceImpl` which logs `partner_user_id` and `program_id` but not SSN).
- **NACHA / Reg E**: ACH withdraw path (`WithdrawType.ACH`) creates bank account devices and submits withdraw actions to Order Service. Void capabilities exist for both ACH and Check instruments.
- **OFAC / AML**: Recipient screening is invoked in `CreateAccountService.performRecipientScreening()` using `AccountHelper.performRecipientScreening()` after account creation. Screening failure is caught and logged but does not fail the transaction — this is a compliance gap (see Business Risks).
- **GDPR/CCPA**: PII (address, email, phone, name, DOB, SSN) transits through the API and is forwarded to Order Service and cbase systems. No data masking or explicit retention controls are visible in this codebase.

## Business Risks

1. **Recipient screening non-blocking**: `performRecipientScreening()` in `CreateAccountService` is wrapped in a broad `catch(Exception e)` that only logs the error and continues, meaning OFAC screening failures do not block account creation. This is a significant AML/OFAC compliance risk.
2. **Test Mode in production code**: `AccountManagementApiServiceImpl` contains a `TestAPI` bean that can simulate delays, return failures, or return processing states. If `testAPI.isTestMode()` is enabled in a non-dev environment, it could cause production disruption. No environment guard beyond configuration is present.
3. **CVV logged in debug**: `CvvInquiryService` line 64 logs `">>> Retrieved Shared Secret : " + profile.getSharedSecret()` at DEBUG level. If DEBUG logging is enabled on `com.citi` package in production, shared secrets appear in logs.
4. **Hardcoded JWE secret key**: `accountmanagementapi.yaml` line 90 contains `jwe.secretKey: '$C&F)J@NcRfUjWnZr4u7x!A%D*G-KaPd'` as a default that is meant to be overridden from app config but is committed to source code. If app config override is absent, this default secret protects DDA encryption.
5. **`allow-circular-references: true`**: `application.yml` line 36 forces circular bean dependency resolution, indicating architectural coupling issues not resolved in the Boot migration.
6. **No transaction idempotency timeout documented**: `ProcessSweepResponse` with `PROCESSING` status can be returned on JMS timeout. Clients must poll `getRequestStatus`, but no maximum retry guidance or dead-letter handling is exposed at the API layer.
