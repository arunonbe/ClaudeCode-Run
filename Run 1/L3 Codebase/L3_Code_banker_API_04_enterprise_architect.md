# banker_API — Enterprise Architect View

## Platform Generation (Gen-1/Gen-2/Gen-3)

**Assessment: Gen-2 (mid-generation, partially modernized)**

Evidence for Gen-2 classification:
- **Transport layer is SOAP/Apache Axis 1.4** — a 2006 vintage SOAP framework. The WSDL was generated by "Apache Axis version: 1.4, Built on Apr 22, 2006." This is a definitive Gen-1 integration pattern.
- **Containerized** — The service has been migrated to Docker/AKS. A `Dockerfile` and Kubernetes rollout config exist. This indicates a Gen-2 lift-and-shift modernization.
- **Java 21** — The JVM target is Java 21 (current LTS), but the programming model (XML Spring context, static singleton, JAX-RPC) is pre-Spring-Boot, pre-annotation configuration.
- **No REST API** — Zero REST endpoints exist in application logic. The single REST endpoint is the health check (`GET /hc`). All business operations are SOAP only.
- **No Spring Boot** — Spring Framework is used directly via XML bean configuration. No `@SpringBootApplication`, no embedded server, no actuator.
- **Lombok present** — `@Slf4j` is used, indicating some modern tooling adoption.
- **Jakarta EE 6.0 Servlet spec** — `web.xml` declares `version="6.0"` (Jakarta EE 10), indicating the servlet container has been modernized even though the application code uses legacy patterns.

The service represents a legacy SOAP service that has been containerized and migrated to AKS without being re-architected.

## Business Domain

**Financial Controls / Disbursement Authorization**

Banker belongs to the **Payments Orchestration** domain. It is the financial gatekeeper for all outbound disbursement operations. In Onbe's value chain:

- **Upstream**: Job Service (batch file processing), Order Service (API-driven orders), ClientZone (web portal orders)
- **Downstream**: Great Plains (Microsoft ERP/general ledger), eCountCore profile service, notification service

Banker does not move money. It authorizes the movement of money by asserting that sufficient program funds exist before a disbursement record is committed.

## Role in Platform

Banker serves three roles:

1. **Pre-authorization gate**: Every disbursement job or order must call `auth()` before funds are committed. Without a successful authorization, downstream processing cannot proceed.

2. **Fund reservation ledger**: Banker maintains its own shadow ledger (`banker_reserved_sources`) that tracks uncommitted (authorized but not yet settled) fund commitments. This ledger reconciles against GP's actual posted balances.

3. **Escalation trigger**: When an authorization request exceeds a user's role-level limit, Banker triggers an approval workflow by calling `sendApprovalNotification()`, notifying higher-level users to manually approve.

This makes Banker a **critical path dependency** for all Onbe disbursement operations. A Banker outage blocks all job and order processing.

## Dependencies

### Inbound (consumers of Banker)
- **Job Service** — Calls `auth()` / `unAuth()` / `cancelReservedSource()` / `forceSettleReservedSource()` for batch file processing
- **Order Service (ClientZone)** — Calls `auth()` for API-driven orders
- **Other services** — Any service needing `getAvailableFunds()`, `getProgramInfo()`, `getFinanceBalances()` for reporting

Consumers use the `banker-common` JAR (`BankerServiceAPI` interface + client Spring XML) distributed via GitHub Packages.

### Outbound (Banker depends on)

| Dependency | Technology | Coupling Level |
|---|---|---|
| Director service | Internal service discovery | Hard — no datasources without it |
| Banker DB (SQL Server) | JDBC + Spring stored procs | Hard — owns the reserved-sources ledger |
| User DB / cbaseapp (SQL Server) | JDBC + Spring stored procs | Hard — every auth call validates user |
| Great Plains (SQL Server, multiple instances) | JDBC + Spring stored procs, regex routing | Hard — all financial data lives here |
| eCountCore / xplatform | Internal Java library (`com.cbase.*`, `com.ecount.*`) | Hard — profile service calls for currency, labels, notifications |
| Apache Axis 1.4 (Jakarta port) | SOAP framework | Hard — transport layer |
| Spring Framework (not Boot) | IoC / AOP / JDBC | Hard — entire wiring |

### Shared Artifacts

- `request-context` (`com.citi.prepaid.module`) — Request context propagation

## Integration Patterns

| Pattern | Implementation | Location |
|---|---|---|
| **SOAP/RPC** | Apache Axis 1.4, JAX-RPC, `BankerServiceAPIImplService` / `bankerServiceAPI` port | `wsdl.xml`, `bankerServiceAxis-client.xml`, `web.xml` |
| **Stored procedure delegation** | All DB access via Spring `StoredProcedure` subclasses or `JdbcTemplate.query()` | `banker-impl/src/main/java/...dao/...` |
| **Registry-pattern datasource routing** | `ProgramStoredProcedureFactory`: regex map of program → GP datasource, loaded from DB | `ProgramStoredProcedureFactory.java` |
| **Singleton service manager** | `BankerServiceManagerImpl.getInstance()` — double-checked null singleton | `BankerServiceManagerImpl.java` line 132–136 |
| **AOP audit/exception wrapping** | Spring AOP pointcuts on all `BankerServiceManager` methods | `banker-core.xml` lines 48–68 |
| **Resilient client proxy** | `ReloadableBean` wraps the Axis client; falls back to `NoopBankerService` if service is unreachable | `bankerService-client.xml` |
| **Transactional isolation** | Spring `DataSourceTransactionManager` + `SERIALIZABLE` isolation on Banker DB writes | `banker-transaction.xml` |
| **Command/Action pattern** | Each business operation is encapsulated in a `BankerServiceAction` subclass (`Authorize`, `UnAuthorize`, `GetAvailableFunds`, etc.) | `banker-impl/src/main/java/...core/action/` |

## Strategic Status

**Status: Active, Critical, Legacy Architecture — Modernization Required**

- Banker is on the **critical path** of all disbursement processing. It cannot be decommissioned without a replacement.
- The SOAP/Axis interface is the primary technical debt item. No new services in Onbe's stack should be consuming SOAP APIs.
- The deep coupling to Great Plains (Microsoft GP) is an architectural constraint. GP acts as the system of record for program financial data. Any migration away from Banker must either retain GP integration or migrate the financial data platform simultaneously.
- The `banker-common` JAR is distributed to consumers as a client library. Any API changes require coordinated version releases across all consuming services.
- The `banker-tester` module is commented out in the parent POM, suggesting the testing tool is no longer maintained.
- The service was previously deployed on Windows bare-metal / VMs (GitLab CI targeting `d-na-app03`, `q-na-app03`). It has been containerized and moved to AKS. The GitLab CI config remains in the repo as a historical artifact.
- PACT contract testing is configured (`PACT_PACTICIPANT: banker-api`) but provider verification is disabled (`VERIFY_PROVIDER_PACT: false`), meaning contract drift may go undetected.

## Migration Blockers

| Blocker | Impact | Notes |
|---|---|---|
| Apache Axis 1.4 SOAP interface | High — all consumers use SOAP/JAX-RPC | Migrating to REST requires all consumers (Job Service, Order Service, ClientZone) to be updated simultaneously or a SOAP-to-REST façade introduced |
| `banker-common` JAR distribution | High — consumers embed the client JAR | Any interface changes require coordinated releases |
| Microsoft Great Plains dependency | High — all financial data is in GP | GP is the system of record; migration requires either GP API abstraction layer or GP replacement |
| Static singleton with in-memory caches | Medium — affects horizontal scaling | In a multi-pod AKS deployment, each pod has its own in-memory cache. Cache invalidation (e.g., after `updateProgramExpressionsDatasourceNames()`) only refreshes the pod that received the call |
| `SERIALIZABLE` transaction isolation | Medium — limits throughput | Acceptable for current volume; may need revisiting for scale |
| No automated tests in CI | Medium — regression risk | Test infrastructure exists but `maven.test.skip` is hardcoded in all pipelines |
| Spring XML configuration (no Spring Boot) | Low-Medium — migration effort | The entire Spring context is XML-driven. Migration to Spring Boot would require significant refactoring |
| XStream serialization in production code | Low — CVE exposure | XStream used in `LoggingUtil.toXML()` which is called by all DTO `toString()` methods; suppressed CVEs need formal risk acceptance |
| `jakarta.servlet.http.HttpUtils.java` | Low — shim class | `banker-service/src/main/java/jakarta/servlet/http/HttpUtils.java` exists as a compatibility shim, indicating direct classpath manipulation for legacy Axis compatibility |
