# correlation-core_LIB — Business Analyst View

## Business Purpose

`correlation-core` is a shared Java library (JAR) that provides **correlation ID lifecycle management** across Onbe's prepaid/payments platform services. Its sole purpose is to generate, propagate, and preserve a unique request-level identifier as work crosses thread boundaries, HTTP hops, and JMS message channels. This enables end-to-end traceability for every business transaction — a non-negotiable requirement in a PCI DSS Level 1 / SOC 2 regulated environment where audit trails for cardholder transactions must be complete and tamper-evident.

- Artifact: `com.ecount.opensource:correlation-core:2.0.1`
- Parent BOM: `com.parents:prepaid-parent:6.0.12`
- Language / Runtime: Java 21
- Packaging: JAR (shared library, not a deployable service)

## Business Capabilities

| Capability | Class | Description |
|---|---|---|
| New correlation ID generation | `CorrelationIDContext.requiresNew()` / `requiresNew(String)` | Creates a UUID-based or caller-supplied ID and binds it to the current thread via SLF4J MDC |
| Correlation ID reuse | `CorrelationIDContext.requires()` | Returns the existing thread-local ID if present; creates one if absent — ensures every thread has an ID |
| Thread-local cleanup | `CorrelationIDContext.clear()` | Removes the ID from MDC to prevent context bleed between requests (thread-pool reuse scenarios) |
| Async propagation — Callable | `CorrelationCallable<T>` | Captures the parent thread's ID at instantiation; restores it in the child thread before `doCall()` executes |
| Async propagation — Runnable | `CorrelationRunnable` | Same pattern as `CorrelationCallable` for `Runnable` tasks |
| ExecutorService decoration | `CorrelationExecutorServiceDecorator` | Drop-in wrapper for any `ExecutorService`; automatically wraps all submitted tasks with correlation-aware wrappers |
| ThreadFactory decoration | `CorrelationThreadFactoryDecorator` | Wraps a `ThreadFactory` so threads created via it inherit the caller's correlation ID |
| Constants for cross-channel headers | `LogContextConstants` | Defines canonical key names for MDC (`correlationId`), HTTP header (`CORRELATION-ID`), and JMS header (`APP_CorrelationID`) |

## Business Entities

- **CorrelationID** — a value object (`Serializable`, `equals`/`hashCode` based on string value) representing a single correlation token. Package-private constructor enforces creation only through `CorrelationIDContext`.
- **Correlation ID string value** — a UUID (random, version 4) generated by `UUID.randomUUID()` unless a caller-provided value is used (e.g., inbound HTTP header).

## Business Rules & Validations

1. **Every thread must have a correlation ID.** `CorrelationIDContext.requires()` auto-generates one if none exists; no downstream code should ever receive a null ID.
2. **Caller-supplied IDs take precedence.** `requiresNew(String id)` and `requiresNew(CorrelationID id)` allow inbound IDs (from HTTP or JMS headers) to be injected, preserving the chain from upstream callers.
3. **IDs must be cleared after use.** `CorrelationCallable.call()` and `CorrelationRunnable.run()` both call `CorrelationIDContext.clear()` after execution completes, preventing stale IDs from leaking into recycled pool threads.
4. **invokeAll / invokeAny bypass decoration.** `CorrelationExecutorServiceDecorator.invokeAll()` and `invokeAny()` delegate directly to the underlying `ExecutorService` without wrapping tasks — callers that use these methods will not have correlation IDs automatically propagated (see Business Risks).
5. **ID format is opaque to consumers.** `CorrelationID` has a package-private constructor; external code can only receive instances via `CorrelationIDContext`, enforcing consistent generation.

## Business Flows

```
Inbound HTTP/JMS request
  └─ Filter/Interceptor (consumer service, not in this lib)
       └─ CorrelationIDContext.requiresNew(headerValue)   ← inject inbound ID
            └─ Business logic on main thread
                 └─ CorrelationExecutorServiceDecorator.submit(task)
                      └─ CorrelationCallable wraps task, captures ID
                           └─ Worker thread: restores ID → executes task → clears ID
```

The library does not contain HTTP filters or JMS listeners itself; it provides the primitives that those cross-cutting concerns rely on. `LogContextConstants` publishes the agreed header names so both producers and consumers use identical keys.

## Compliance & Regulatory Concerns

- **PCI DSS v4.0.1 — Req 10 (Log and Monitor):** Correlation IDs embedded in every log line via SLF4J MDC enable reconstruction of the full activity trail for a cardholder transaction across microservices, satisfying audit-log completeness requirements.
- **SOC 1 / SOC 2 — Logging controls:** Consistent, tamper-evident correlation of log events across services supports the operational evidence required for SOC audit procedures.
- **NACHA / Reg E — Transaction traceability:** For ACH and prepaid disbursements, end-to-end traceability from initiation to settlement is required for dispute resolution. This library provides the foundational mechanism.
- **NIST CSF 2.0 — Detect / Respond:** Correlated logs are prerequisite to effective anomaly detection and incident response. Without correlation IDs, cross-service attack patterns (e.g., fraudulent payout requests) are difficult to reconstruct.
- **GDPR / CCPA — Data subject request fulfillment:** Being able to correlate all log records for a single request is important when locating all processing records for a data subject access or deletion request.

## Business Risks

1. **invokeAll / invokeAny not decorated** — Tasks submitted via these two methods do not inherit the caller's correlation ID. Any business logic using these methods produces log lines with missing or wrong IDs, breaking audit trails (PCI DSS Req 10 risk).
2. **No ID validation** — Caller-supplied IDs (from `requiresNew(String id)`) are stored as-is; there is no length check, format enforcement, or sanitisation. A maliciously crafted or excessively long string could pollute log output (log injection risk).
3. **Silent ID generation in unexpected places** — `CorrelationIDContext.requires()` silently mints a new UUID if none exists, which can mask missing propagation by making log lines appear correlated when they are not.
4. **Exception path in CorrelationCallable** — If `doCall()` throws an unchecked exception, `CorrelationIDContext.clear()` is never called (no finally block), leaving a stale ID in the thread. This is a thread-pool safety issue with potential to associate subsequent requests with the wrong ID.
