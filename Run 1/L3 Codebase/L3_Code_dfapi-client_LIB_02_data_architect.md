# Data Architect Report — dfapi-client_LIB

## 1. Data Stores

dfapi-client_LIB is a **library** with no persistent data stores of its own. All data flows transiently in-memory and via external protocol calls.

| Storage Type | Location | Purpose |
|---|---|---|
| In-memory | `QuoteRequest` / `QuoteResponse` Java objects | Request/response DTOs |
| IBM MQ queue (HTTP/JMS path) | `CPS.DF.RQ1` on `GU00` queue manager, host `192.193.99.44:27197` | JMS transport queue for DFAPI requests |
| XML string (JMS path) | In-memory `String` | Serialised request/response for MQ transport |

## 4. Encryption

| Mechanism | Status | Detail |
|---|---|---|
| TLS (HTTP transport) | Technically enabled | `https://citigroupsoa.citigroup.com/` endpoint uses HTTPS |
| SSL certificate validation | **Disabled** | `TrustAllSSLSocketFactory.java` in `dfapiclient-impl` — bypasses all cert verification |
| MQ channel encryption | Unknown | `CPS.TOMCAT.SVRCONN.U` channel; encryption not visible in config |
| JAXB XML payload | Plaintext XML | No message-level encryption |
| Client certificate | Referenced in `httpclient.properties` (First National Bank of Omaha cert details) | Certificate-based auth for Citi endpoint |

**Critical Gap**: `TrustAllSSLSocketFactory` must never be used in production. Its presence in the `dfapiclient-impl` module means any caller using the HTTP handler trusts any TLS certificate, enabling MITM attacks.

## 5. Data Flow

```
Calling Service
  │ QuoteRequest (Java object)
  ▼
DFAPIClientImpl.execute()
  │
  ├─ [HTTP path]
  │    HTTPHandler → Apache Axis SOAP stub
  │      → HTTPS POST to citigroupsoa.citigroup.com/DFAPIService/DFAPIWSDLServ
  │      ← SOAP response
  │    → Convert to QuoteResponse
  │
  └─ [JMS path]
       JMSHandler → XMLHandler.getXML() → XML string
         → MQJMS.executeSendMessage() → IBM MQ GU00/CPS.DF.RQ1
         → MQJMS.executeReceiveMessage(correlId) ← XML string
         → XMLHandler.getResponse() → QuoteResponse
```

## 6. Data Quality

- **Date handling**: `QuoteRequest.getReleaseTime()` and `getValueDate()` use `SimpleDateFormat` without thread safety (`DateFormat` is not thread-safe). Concurrent access to a shared `QuoteRequest` instance could cause incorrect date parsing.
- **Null handling**: `HTTPHandler.getDFAPIResponse` wraps conversion in try/catch but silently continues on `ParseException`; partially populated `QuoteResponse` returned without error indication.
- **`returnCode`/`returnMessage`**: Response success/failure must be checked by callers; no exception thrown on non-success `returnCode`.

## 7. Compliance Gaps

| Gap | Standard | Detail |
|---|---|---|
| `TrustAllSSLSocketFactory` — disables TLS cert validation | PCI DSS Req 4.2.1 | MITM vulnerability on Citi DFAPI connection |
| Internal network hostnames in committed test resources | PCI DSS Req 12.3 | `dflnxswapu.nam.nsroot.net`, `192.193.99.44`, `webproxy.ssmb.com` in version control |
| Apache Axis 1.4 EOL | PCI DSS Req 6.3 | No security patches for end-of-life SOAP library |
| No response signature verification | Dodd-Frank / data integrity | Response XML not cryptographically verified |
| `SimpleDateFormat` not thread-safe | Data integrity | Date parsing errors under concurrency |
