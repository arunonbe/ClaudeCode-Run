# Data Architect View — js-import_SVC

## Overview

`js-import_SVC` is a data-ingestion service. Its core data responsibility is to parse a proprietary fixed-width hierarchical file format and fan out each logical record into a normalised relational schema within the `jobsvc` SQL Server database. Understanding the data model, field constraints, and persistence paths is essential to assessing data quality, lineage, and compliance posture.

## Input Data Format

The JS file format is a fixed-width, line-oriented, record-type-discriminated binary-ish flat file. Each line begins with a record type byte that the `JobSvcParserLineLexer` uses to dispatch to the appropriate `RecordLine` subclass. The `Parser` class (`com.ecount.datagen.Parser`) maintains a state machine over these types.

| 4 | `BatchFooter` | Batch trailer with record count |
| 5 | `Request` | Per-payee wrapper |
| 7 | `CreateAccountsRequest` | Cardholder create |
| (UA) | `UpdateAccountRequest` | Cardholder profile update |
| (AF) | `AddFundsRequest` | Fund load |
| (W) | `WithdrawRequest` | ACH / check withdrawal |
| (SP) | `SpinPaymentRequest` | Balance conversion payment |
| (ST) | `StopPaymentRequest` | Stop payment |

| (CWU) | `CreateWebUserRequest` | Portal user create |
| (UWU) | `UpdateWebUserRequest` | Portal user update |
| (SL) | `SetLocationCodeRequest` | Location restriction |
| (SILA) | `SetInvLocationAttributesRequest` | Inventory location attribute |

### Field Typing

Fields are defined via `com.ecount.datagen` types:
- `CharType` — fixed-length character, padded with spaces. Example: `FIRST_NAME` (25), `HOMEEMAIL` (50).
- `NumericType` — fixed-length numeric, zero-padded.
- `RecordIdType` — 8-character surrogate ID generated by `IDGeneratorImpl`.
- `PadCharsType` — reserved filler bytes.
- `CurrencyType` — monetary amounts.
- `DateType` — date fields.

The `ValidDataTypesContainer` (injected from `service.properties` at `D:/c-base/config/service/jobservice/JSImporter/service.properties`) controls which character sets are valid for `CharType` fields — enabling environment-specific (domestic vs EMEA) character validation.

## Persistence Layer (DAO Map)

Each record type maps directly to a dedicated DAO class that executes a stored procedure or SQL against `jobsvc`:

| DAO | Target Table / SP | Key PII Fields Written |
|---|---|---|
| `CreateAccountRequestDAO` | jobsvc.create_account_request | FIRST_NAME, LAST_NAME, EMAIL, PHONE, ADDRESS |
| `CreateAccountsSecureInfoAddendaDAO` | jobsvc.create_accounts_secure_info_addenda | SSN, other SAD |
| `CreateExtendedAccountRequestDAO` | jobsvc.create_extended_account_request | Extended KYC |
| `AddFundsRequestDAO` | jobsvc.add_funds_request | Amount, PromoID |
| `WithdrawRequestDAO` | jobsvc.withdraw_request | Bank account / routing (ACH), check data |
| `SpinPaymentRequestDAO` | jobsvc.spin_payment_request | Payment amount, target |
| `CreateWebUserRequestDAO` | jobsvc.create_web_user_request | Username, email |
| `EmailNotificationRequestDAO` | jobsvc.email_notification_request | Email address, template ref |
| `InsertJobFileDAO` | jobsvc.job_file | File metadata |
| `InsertJobBatchDAO` | jobsvc.job_batch | Batch metadata |

All DAOs are Spring beans wired in `jobsvc_import.xml` against the `JobSvcDataSource` JNDI connection pool.

## ID Generation

`IDGeneratorImpl` (`com.ecount.service.idgenerator.IDGeneratorImpl`) maintains numeric ID sequences stored in `jobsvc` tables. The `GetIDSequenceDAO` / `UpdateIDSequenceDAO` / `UpdateIDSequenceExhaustedDAO` DAOs manage a compare-and-swap approach to sequence increment. This is not a UUID strategy — it is an integer sequence that wraps at exhaustion, which creates collision risk under high-volume parallel imports.

## Data Lineage

```
Client SFTP / HTTP POST
  → JobValidatorServlet.java
    → JSContext (Spring XML bootstrap, jobsvc_import.xml)
      → RequestFileParser (state machine)
        → JobSvcParserLineLexer (line tokeniser)
          → RecordLine subclasses (typed field objects)
            → InsertRecordsDAOManager
              → Individual DAO (SP / SQL)
                → jobsvc SQL Server tables
                  → jobservice_SVC (downstream consumer)
```

## Data Quality Controls

- Field-length enforcement is built into `CharType` / `NumericType` constructors — values exceeding declared lengths are rejected at parsing time.
- `ValidDataTypesContainer` enforces character-set rules per environment, preventing injection of non-ASCII or control characters via the char-type validation.
- `ParserRule` state machine enforces record sequence grammar — disallowing invalid action-addenda combinations (e.g., QC 9661 fix at `RequestFileParser.java` line 128 prevents duplicate `CreateAccountsRequest` records under the same `Request`).
- There is no apparent field-level format validation for email addresses or phone numbers within the parser — values are accepted as `CharType` strings and written verbatim to the database.

## Data Retention and Disposal

The service does not implement any retention or disposal logic. Records are persisted to `jobsvc` and ownership passes to the Job Service and database tier. Retention periods must be governed by database-level policies and the broader enterprise data governance framework.

## Key Data Risks

1. **No field-level encryption at rest** — PII and financial fields are stored in plaintext in `jobsvc` tables.
2. **Flat-file transmission** — if the upstream SFTP layer is misconfigured, files containing PII/SAD could be exposed in transit.
3. **Integer ID collision** — the `IDGeneratorImpl` sequence mechanism could produce collisions under high concurrency or after sequence exhaustion.
4. **No masking in error logs** — the `ErrorLogger` class (`util/ErrorLogger.java`) does not mask PII before writing errors; field values from failed records may appear in application logs.
5. **Legacy log4j 1.x** — `pom.xml` declares `log4j:log4j:1.2.17` (provided scope). Log4j 1.x reached end-of-life and has known critical CVEs (CVE-2019-17571 socket server deserialization). Even as `provided`, the container must not expose this as active.
