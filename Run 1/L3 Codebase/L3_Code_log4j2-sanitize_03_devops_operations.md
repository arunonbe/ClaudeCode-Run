# DevOps and Operations View — log4j2-sanitize

## Build System

`log4j2-sanitize` is a Spring Boot 3.3.5 Maven project. Key build configuration (`pom.xml`):

- **Java version**: 21 (`<java.version>21</java.version>`)
- **Spring Boot parent**: `3.3.5`
- **Maven Wrapper**: bundled (`.mvn/wrapper/maven-wrapper.properties`)
- **Packaging**: JAR (default Spring Boot)
- **Group ID**: `com.onbe.logging`
- **Artifact ID**: `log4j2-sanitize`
- **Version**: `0.0.1-SNAPSHOT`

The project excludes the default Spring Boot logging starter (`spring-boot-starter-logging`) and replaces it with `spring-boot-starter-log4j2` — this is the correct pattern for using Log4j2 with Spring Boot:

```xml
<exclusion>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-logging</artifactId>
</exclusion>
...
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-log4j2</artifactId>
</dependency>
```

`commons-text:1.12.0` is the single custom dependency beyond the Spring Boot BOM.

Build command:
```sh
./mvnw clean package
```

## CI/CD Pipeline

No CI/CD pipeline configuration files are present (no `.gitlab-ci.yml`, no `.github/workflows/`). This is consistent with the repo being a reference/proof-of-concept, not a deployable service. Before this pattern is adopted into production services, a CI pipeline should be created that:

1. Builds and runs unit tests
2. Performs static analysis (e.g., SpotBugs, OWASP Dependency Check)
3. Publishes the JAR to the internal Nexus/Maven repository if promoted to a shared library
4. Tags releases

## Deployment Model

As a proof-of-concept, this is not independently deployed. The intended deployment model is:

1. **Adopt as a shared library**: The `SanitizingFilter` class and `log4j2-spring.xml` configuration should be extracted into a standalone Maven library artifact published to Onbe's internal Nexus repository.
2. **Consume in target services**: Each Spring Boot service on Log4j2 adds the library as a dependency and includes the `log4j2-spring.xml` configuration.
3. **Spring Boot auto-configuration**: Spring Boot auto-discovers `log4j2-spring.xml` on the classpath when Log4j2 is active.

## Testing

The test class `Log4j2SanitizeApplicationTests.java` (`src/test/java/com/onbe/logging/Log4j2SanitizeApplicationTests.java`) contains only the Spring Boot context load test (auto-generated by Spring Initializr). There are **no unit tests** for the `SanitizingFilter` itself. This is a critical testing gap:

- No test verifying that `<script>alert('XSS')</script>` is escaped to `&lt;script&gt;alert(&#39;XSS&#39;)&lt;/script&gt;`
- No test verifying PAN is not inadvertently affected (or should be masked — see Data Architect notes)
- No test verifying CRLF behaviour
- No test verifying exception handling if `MutableLogEvent` cast fails

The `LogConfig.java` (`src/main/java/com/onbe/logging/LogConfig.java`) demonstrates the filter in a `@PostConstruct` method, but this is an integration smoke test, not a unit test of the filter logic.

## Operational Considerations

As a proof-of-concept, there are no specific operational runbooks. However, teams adopting this pattern should be aware of:

1. **Performance overhead**: `StringEscapeUtils.escapeHtml4()` is a string scan on every log message. Under high-throughput logging (e.g., TRACE-level logging in production), this could add measurable latency. Async Log4j2 appenders (`AsyncAppender`) can mitigate this.

2. **Filter placement**: The filter is attached to the `Console` appender only. Any `File`, `RollingFile`, or `Socket` appenders added to production configurations must also include the `<SanitizingFilter/>` element.

3. **Log level threshold ordering**: In `log4j2-spring.xml` (lines 22–23), `SanitizingFilter` runs before `ThresholdFilter`. This means the sanitisation is applied even to messages that would be discarded by the threshold filter — a minor efficiency issue.

4. **MutableLogEvent availability**: The cast to `MutableLogEvent` (`SanitizingFilter.java` line 24) assumes synchronous logging. In async appender configurations with disruptor-based Log4j2, the event may be a different implementation. A `ClassCastException` here would suppress the log event without notice. A `instanceof` check before the cast is recommended:
   ```java
   if (event instanceof MutableLogEvent mutableLogEvent) {
       mutableLogEvent.setMessage(new SimpleMessage(sanitizedMessage));
   }
   ```

5. **Spring Boot profile awareness**: The `application.properties` file contains only `spring.application.name=log4j2-sanitize`. The `log4j2-spring.xml` is the active Log4j2 configuration (Spring Boot auto-detects `-spring` suffix for profile-aware log configs).
