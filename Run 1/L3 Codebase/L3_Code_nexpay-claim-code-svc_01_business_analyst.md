# nexpay-claim-code-svc — Business Analyst View

## Business Purpose

`nexpay-claim-code-svc` is a **Gen-3 NexPay microservice** that manages the lifecycle of **claimable payments** — prepaid disbursements issued to recipients who must redeem them by presenting a unique claim code. It provides the data layer and API for claim code lookup, validation, and recipient registration, supporting multi-rail payout choices (virtual card, physical card, push-to-card, PayPal, ACH, check).

The service bridges the NexPay orchestration platform and the legacy eCount core: it reads from a database that is synced or co-managed with `EcountCore..claimable_payment`, and stores recipient registration details for the Gen-3 payment flow.

## Capabilities

- **Claim code lookup**: Retrieve full payment details for a given claim code (`GET /claimable/{claimCode}`)
- **Claim code validation**: Validate one or more claim codes against business rules (status, expiration) in a single batch request (`POST /validateCodes`)
- **Recipient registration lookup**: Retrieve recipient personal and address information by UUID (`GET /recipient-registrations/{id}`)
- **Audit trail**: Every change to claimable payment data is tracked via Hibernate Envers with actor identity sourced from OpenTelemetry baggage
- **Database schema management**: Flyway versioned migrations manage the `nexpay_claimable` SQL Server database schema

## Business Rules

1. **Claim code validation order** (mirrors `dbo.validate_claimable_payment` SP):
   - If status = `claimed` → error: `"Already_Claimed"`
   - If status ≠ `issued` (including blocked, canceled, expired) → error: `"invalid_status_code"`
   - If `expirationDate` is non-null and in the past → error: `"Expired"`
   - Otherwise → valid
2. Claim codes must be globally unique (`UNIQUE` constraint on `claim_code` column)
3. `RecipientRegistration.id` is a UUID assigned externally (not auto-generated by the service) — the table uses `@Id` without `@GeneratedValue`
4. `ClaimablePayment.amount` is stored as an integer (cents/smallest currency unit)
5. `ClaimablePayment` entity is marked `@Immutable` on its status and modality lookup sub-entities — lookup tables are read-only
6. Flyway is **disabled** in the QA profile (`flyway.enabled: false`) because the QA DB user lacks DDL permissions; schema is pre-provisioned

## Flows

1. **Claim flow** (external caller → service → DB):
   - Caller presents `claimCode`
   - `GET /claimable/{claimCode}` → `ClaimablePaymentRepository.findByClaimCode` → return `ClaimablePaymentDetail` DTO
   - On not-found: HTTP 404

2. **Validation flow**:
   - Caller submits list of claim codes
   - `POST /validateCodes` → bulk `findByClaimCodeIn` → per-code validation logic → list of `ClaimCodeValidationResult`

3. **Recipient registration lookup**:
   - `GET /recipient-registrations/{id}` → `RecipientRegistrationRepository.findById` → return `RecipientRegistrationDetail` DTO

4. **Audit flow**:
   - Any entity mutation → Hibernate Envers intercepts → `CustomRevisionListener` extracts `actor.id` from OTel baggage → stores in `revinfo.actor_id`

## Compliance Relevance

- Stores recipient PII: full name, email, home/business/mobile phone, full mailing address — **CCPA, GDPR, and Reg E relevant**
- `dda` field in `claimable_payment` is a DDA (demand deposit account) identifier — **PCI DSS / banking data** relevance; if this maps to a card DDA it may be in scope
- `ownerId` UUID links to an eCount cardholder account — **CDE-adjacent**
- Claim codes are single-use financial instruments — must not be logged in plain text
- Hibernate Envers audit trail with actor ID helps satisfy PCI DSS Req. 10 (audit logging)

## Risks

- `claimCode` appears in log messages at INFO level (`log.info("getClaimable: called for claimCode={}", claimCode)`) — claim codes are sensitive single-use financial tokens that should not appear in logs
- `dda` field purpose is not fully documented in source; if it represents a card DDA, it is a PCI DSS-sensitive identifier
- Seed data in `V5__create_recipient_registration.sql` includes a real Onbe employee email address (`andrew.smirnoff@onbe.com`) — production migrations should not contain real PII
- `RecipientRegistration` has no `created`/`updated` timestamps — no way to audit when records were inserted
