## Business Purpose

The nexpay-ordervalidator-svc is the gatekeeping service for disbursement orders. Its mission is to evaluate whether a proposed disbursement transaction meets all pre-execution business rules before the order is routed to the fulfillment orchestrator. By centralising validation logic in a dedicated service, Onbe ensures consistent application of business controls — amount limits, velocity rules, eligibility criteria, and regulatory screening — regardless of which client application or payment rail initiated the disbursement.

In a PCI DSS Level 1 payments environment, the validator acts as a first-line control that reduces the risk of fraudulent, erroneous, or prohibited transactions reaching the card processor and becoming real financial movements.

## Business Capabilities

### Non-Monetary Validation

Implemented in `NonMonetaryValidationService.processCardCreationValidation()`. This validation evaluates structural and eligibility criteria that do not depend on the actual payment amount:
- Transaction ID presence and format
- Account ID validity
- Program ID existence
- Card type eligibility
- Timestamp validity

The service creates a `CardCreationRequest` object, invokes `NonMonetaryValidationService`, and returns a `ValidationCompleted` response with a `PASSED` or `FAILED` result and optional error message.

### Funds Validation

Implemented in `FundsValidationService.processFundsValidation()`. This validation evaluates the financial aspects of a requested disbursement:
- Transaction ID presence (required)
- Validation type must be `FUNDS`
- Requested amount must not exceed the configured `FUND_LIMIT` (currently hardcoded at `1000.0` in `FundsValidationService.java` line 16)
- Amount must be a valid numeric format

**Critical observation**: The `FUND_LIMIT` of `1000.0` is a hardcoded constant in the source code. For a production payments service, transaction amount limits must be configurable without a code deployment, typically sourced from Azure App Configuration or program configuration. A hardcoded limit cannot be adjusted to respond to fraud events, regulatory changes, or program-specific rules without a full build-and-deploy cycle.

**Also critical**: The code comment at `FundsValidationService.java` line 1 reads `// Code generated by Claude` and `ValidationService.java` line 19 reads `// Code updated by Claude`. These comments indicate that portions of this service were AI-generated rather than authored by the engineering team. AI-generated code in a PCI DSS compliance-critical validator requires enhanced peer review before production certification.

### Address Verification Integration

The service includes an `AddressVerificationMapper` and references an address verification service at `nexpay-ordervalidator-impl`. The `application.yaml` integration profile (lines 174–179) references the address verification service URL. This capability validates that the payee's address is a legitimate mailing address, which is important for check disbursements and ACH address matching.

## OFAC Screening — Clarification of Responsibility

The nexpay-ordervalidator-svc does **not** directly perform OFAC screening. OFAC/sanctions screening is delegated to the `recipient-screening-api`, which is called from the `nexpay-recipientorchestrator-svc` (Step 1 of the claim-code saga via `RecipientScreeningServiceClient`). The validator's role is transactional business-rule validation (amounts, velocity, eligibility), not identity screening.

This separation of concerns is architecturally correct but creates a sequencing dependency: the validator may clear a transaction while OFAC screening is still pending. The two controls must be verified to be logically sequenced in the overall flow before funds can be released.

## Velocity Controls

As implemented, the `FundsValidationService` enforces only a per-transaction amount limit. **There are no velocity controls** (e.g., maximum transactions per day per recipient, maximum cumulative amount per time window) visible in the current codebase. This is a significant gap for AML and fraud risk management. Velocity controls are typically required to comply with NACHA same-day ACH thresholds, card network limits, and Onbe's internal fraud policy.

**Recommendation**: Velocity controls should be implemented referencing a stateful store (e.g., Redis with TTL-based counters) or a dedicated fraud/risk engine. The current `FUND_LIMIT` of $1,000 provides only a single-transaction cap.

## Validation Request/Response Model

The OpenAPI-generated model includes:
- `ValidationRequest`: transactionId, accountId, validationType (FUNDS | NON_MONETARY), requestedAmount
- `ValidationCompleted`: transactionId, validationType, validationResult (PASSED | FAILED | PENDING), errorMessage, processingTime, validationDetails
- `ValidationDetails`: accountStatus, cardholderVerified, riskScore, availableBalance

The `ValidationDetails` response for non-monetary validation contains hardcoded values (`accountStatus: "ACTIVE"`, `cardholderVerified: true`, `riskScore: "LOW"`, `availableBalance: "1000.00"`) as noted in `ValidationService.java` lines 104–108. These are mock values — the service is not yet integrated with a real account status service for these fields.

## Regulatory Obligations

- **Reg E**: The validator is a pre-authorization control. Its PASSED decision is part of the evidence chain that a disbursement was properly authorised.
- **NACHA**: For ACH disbursements, the amount validation must respect NACHA per-transaction and same-day ACH limits. Current implementation does not distinguish ACH from card transactions.
- **OFAC**: Not directly applicable here — see note above.
- **PCI DSS Req 6.2**: The AI-generated code (`FundsValidationService`, `ValidationService`) must be reviewed and tested under PCI DSS secure development requirements before production.
