# Enterprise Architect View — nexpay-ordervalidator-svc

## Position in the NexPay Gen-3 Architecture

The nexpay-ordervalidator-svc occupies the **pre-transaction control layer**, positioned between order intake and fulfillment. It is one of the first services in the disbursement chain to evaluate whether a proposed transaction is permissible. In the Gen-3 architecture it is a distinct microservice, allowing the validation logic to be independently deployable, testable, and scalable.

```
Order Intake (BFF / client API)
         │
         ▼
nexpay-ordervalidator-svc  ← THIS SERVICE (pre-transaction gate)
         │ PASSED / FAILED
         ▼
nexpay-order-orchestrator
(fulfillment + OFAC screening + issuance)
```

## Separation of Concerns from OFAC Screening

A key architectural decision visible in the codebase is that OFAC/sanctions screening is **not** performed in the ordervalidator. The validator is concerned with transaction-level business rules (amount limits, program eligibility, address format). Recipient-level identity screening (OFAC, AML) is delegated to the `recipient-screening-api` and performed within the orchestration saga (Step 1).

This separation is appropriate because:
1. OFAC screening is identity-centric (about the recipient) while amount validation is transaction-centric.
2. It allows OFAC screening to be performed even for repeat recipients who have already passed amount limits.
3. It allows the validator to be independently swapped or enhanced without affecting screening logic.

However, it creates a sequencing dependency that must be architecturally enforced: order fulfillment must invoke both the validator AND the screening service before releasing funds. The orchestrator currently calls screening but the validator is not called as a blocking gate from the orchestrator's code path — the orchestrator calls the claim-code-svc for validation but not this ordervalidator-svc explicitly. This suggests the ordervalidator may be called from a different entry point (e.g., the BFF or an upstream system) before the orchestrator is invoked.

## API Contract

The service exposes:
- `POST /api/v1/validate` — general validation endpoint dispatching to FUNDS or NON_MONETARY path
- `POST /api/v1/funds-validate` — funds-specific validation endpoint

Both are OpenAPI-generated interfaces implemented by `ValidationApiDelegateImpl` and `FundsValidationApiDelegateImpl` respectively. The service also provides an address validation sub-module (`nexpay-ordervalidator-address`) with its own client API.

## Current Maturity Assessment

| Capability | Maturity |
|---|---|
| Non-monetary structural validation | Partially implemented (mock response details) |
| Funds amount limit | Implemented (hardcoded $1,000) |
| Velocity controls | Not implemented |
| OFAC screening | Not applicable (delegated) |
| Address verification | Partially wired (external client configured) |
| Real-time account status | Not implemented (mocked) |

The service is in an early-to-mid development stage. The AI-generated comment in `FundsValidationService.java` (`// Code generated by Claude`) and `ValidationService.java` (`// Code updated by Claude`) indicates rapid prototyping has occurred but the code has not yet undergone a full engineering review cycle.

## Enterprise Risk Register

| Risk | Severity | Notes |
|---|---|---|
| Credentials committed to source code | Critical | See `application.yaml` dev-test and integration profiles |
| Hardcoded amount limit ($1,000) | High | No runtime configurability; cannot respond to fraud events without deployment |
| AI-generated validation code without formal review | High | PCI DSS secure development lifecycle gap |
| No velocity controls | High | AML/fraud exposure for high-frequency low-amount attacks |
| Hardcoded mock validation details | Medium | Downstream consumers may rely on fabricated `riskScore: LOW` |
| MSSQL integration tests vs PostgreSQL production | Medium | Schema parity risk |
| Duplicate `qa` profile stanzas in application.yaml | Low | Configuration maintenance risk |

## Compliance Position

The validator is a PCI DSS Requirement 6.4 (protect web-facing applications) and Requirement 12.3 (risk management) relevant service. Specifically:

- **PCI DSS Req 6.2.4**: The AI-generated code requires security testing before production use.
- **PCI DSS Req 8**: Committed credentials violate the requirement for secure authentication management.
- **NACHA Rule OR-3**: ACH originators must have velocity controls. The current implementation does not satisfy this for ACH payment rails.

## Recommended Architectural Enhancements

1. **Extract rule configuration**: Replace hardcoded `FUND_LIMIT` with a `ValidationRuleService` that reads per-program limits from `nexpay-config-svc`.
2. **Add velocity control**: Implement a Redis-backed counter service (`VelocityControlService`) checking counts per recipient per time window.
3. **Remove credentials from source**: Use Azure Managed Identity for all environments including dev-test.
4. **Complete address validation integration**: Wire the `AddressVerificationMapper` to the external service and use real response data.
5. **Formal code review**: Complete PCI-compliant peer review of all AI-generated validation logic before production certification.
