## Implementation Architecture

### Delegation Chain

```
POST /api/v1/validate (HTTP request)
    ▼
ValidationApiDelegateImpl
    ├── [if NON_MONETARY] → ValidationService.processNonMonetaryValidation()
    │                           → NonMonetaryValidationService.processCardCreationValidation()
    │                               → [structural rule checks]
    │                               → ValidationCompleted (with hardcoded details)
    │
    └── [if FUNDS] → ValidationService.processFundsValidation()
                         → FundsValidationService.processFundsValidation()
                             → [amount check: <= FUND_LIMIT]
                             → ValidationCompleted
```

### Validation Logic Review

**NonMonetaryValidationService** validates:
- `transactionId` is not null and non-blank
- `accountId` is not null
- `programId` is not `"DEFAULT_PROGRAM"` (the mock value assigned by `ValidationService.java` line 40 is actually `"DEFAULT_PROGRAM"`, which means the programId check always sees a test value in the non-monetary path — this is a logic gap where the BFF's actual programId is discarded)
- `cardType` is not null
- `timestamp` is not null and not in the future

**FundsValidationService** validates:
- `transactionId` not blank
- `validationType` must be `FUNDS`
- `requestedAmount`, if present, must parse as `double` and not exceed `FUND_LIMIT = 1000.0`

The funds validator applies an additive check on amount but does not check for negative or zero amounts, which could allow zero-value disbursements to pass validation.

## Security Architecture

### Credential Leak — Critical Finding

**File**: `nexpay-ordervalidator-boot/src/main/resources/application.yaml`

**Lines 126–127** (dev-test profile):
```yaml
connection-string: Endpoint=https://as-app-configuration.azconfig.io;Id=eSca;Secret=[REDACTED — rotate immediately]
```

**Lines 179** (integration profile):
```yaml
secret: [REDACTED — rotate immediately]
```

Both are live credentials committed to version control. The Azure App Configuration connection string gives full read/write access to that configuration store. The OAuth2 client secret can be used to obtain access tokens for the push-provision API.

**Immediate actions**:
1. Revoke/rotate `Id=eSca` App Config credential.
2. Revoke/rotate OAuth2 client `8106a982-8d61-423e-a32f-08ff30928646` secret `0pf8Q~...`.
3. Replace with `${AZURE_APP_CONFIG_CONNECTION_STRING}` and `${OAUTH2_CLIENT_SECRET}` env vars sourced from Key Vault.
4. Add GitHub secret scanning push protection to prevent future commits.

### Authentication

The service uses Spring Security with Azure APIM as the authentication gateway. Service-to-service calls from the validator to the address verification service use OAuth2 client credentials flow (client ID and secret — currently committed in source; should use managed identity).

## Quality Gaps

### AI-Generated Code

Three files carry explicit `// Code generated by Claude` or `// Code updated by Claude` comments:
- `FundsValidationService.java` (line 1)
- `ValidationService.java` (lines 19, 56)

PCI DSS Requirement 6.2 mandates that bespoke software is developed according to secure coding guidelines and reviewed by qualified personnel. AI-assisted code generation is not prohibited but requires the same review standards as human-authored code. These files must undergo:
1. Security-focused peer review.
2. Unit and integration testing to 100% statement coverage of validation paths.
3. Sign-off by a qualified developer before production certification.

### Hardcoded Mock Values

In `ValidationService.mapToValidationCompleted()` (lines 103–108), the `ValidationDetails` object is populated with values that do not reflect actual account or risk state:
```java
details.setAccountStatus("ACTIVE");
details.setCardholderVerified(true);
details.setRiskScore("LOW");
details.setAvailableBalance("1000.00");
```

If any downstream consumer uses `riskScore` or `cardholderVerified` for routing or reporting, they will consistently receive incorrect data, creating a false confidence in the validation result.

### Zero/Negative Amount Gap

`FundsValidationService.validateFunds()` only checks `amount > FUND_LIMIT`. It does not check for `amount <= 0`. A request with `requestedAmount: "0"` or `requestedAmount: "-50"` will pass validation. The business rule should be `amount > 0 && amount <= FUND_LIMIT`.

## Performance Profile

The service is stateless and CPU-bound (pure rule evaluation). Under Java 25 virtual threads, request handling is extremely lightweight. The only potential latency source is the address verification service call, which is an external HTTP call with OAuth2 token acquisition. Token caching should be implemented using Spring Security's `OAuth2AuthorizedClientManager` with token reuse.
