# Data Architect Report — onbe-spring-boot

## Data Models

`onbe-spring-boot` is a framework library and does not define domain data models or persist business data. It defines configuration data models (Kotlin data classes annotated with `@ConfigurationProperties`) and infrastructure data structures.

Key configuration models:
- `DaprSecretsConfigProperties` — holds the Dapr secret store name and list of secret key names to retrieve. Does not hold secret values directly; values are resolved at runtime from the Dapr sidecar.
- `RestClientConfiguration` — holds HTTP client parameters (timeouts, retry counts, default headers). No sensitive data.

## Sensitive Data Handling

### Secrets in Transit (High Risk Area)
`DaprSecretsConfiguration.init()` calls `DaprClientBuilder().build().use { client -> ... }` to retrieve secrets from the Dapr secret store and injects them into the Spring `ConfigurableEnvironment` as a `MapPropertySource`. The secrets themselves flow through JVM memory as a `MutableMap<String, Any>`. While this is the intended pattern, the secrets map is held in JVM heap and is not explicitly zeroed after use, which means secret values (including database passwords and API tokens) remain in heap until GC collection.

### Log Masking
The `TextUtils.mask()` function (prefixLength=2, postfixLength=2, maxMaskLength=8) is applied before any secret value is logged. This means the actual secret value is never emitted to the log sink. The masking logic reveals at most the first and last 2 characters plus masks up to 8 characters — adequate for observability without exposure.

### PAN/CVV/SSN/DDA
No PAN, CVV, SSN, or DDA fields are defined or processed within this library. These would reside in domain services that consume this framework. The framework provides the masking utility (`TextUtils.mask()`) that domain services should use when logging any sensitive field.

## Database Schema

`onbe-spring-boot-data-r2dbc` provides `ConnectionFactoryFactory` for building R2DBC connection pools for Microsoft SQL Server and PostgreSQL. No schema is defined in the library itself. Test resources include a minimal `schema.sql` (used by the data-r2dbc test module) with a simple table for test validation purposes only — not a production schema.

The library supports:
- MS SQL Server (r2dbc-mssql driver)
- PostgreSQL (r2dbc-postgresql driver)
- Connection pool parameters: initial size, max size, max idle time, max life time, validation query, acquire timeout, create connection timeout

## Data Flows

```
[Dapr Sidecar (Azure Key Vault)] --> DaprSecretsConfiguration.init()
    --> Spring Environment PropertySource (in-memory map)
    --> Consumed by Spring beans via @Value / @ConfigurationProperties

[Downstream Service HTTP Client] --> RetryRateLimitRequestInterceptor
    --> JdkClientHttpRequestFactory (Java HttpClient)
    --> External API endpoint

[Application Database] <--> ConnectionFactoryFactory (R2DBC Pool)
    --> Spring Data R2DBC Repository / DatabaseClient
```

## Encryption and Data Protection Status

- **Secrets at rest:** Delegated to Dapr-configured backend (Azure Key Vault in production). The library itself has no encryption logic.
- **Secrets in transit:** Dapr SDK uses gRPC over localhost loopback for sidecar communication. External Azure Key Vault communication uses TLS via the Dapr sidecar — the library inherits this protection.
- **HTTP connections:** `RestClientConfiguration` defaults include `Accept: application/json` and `Content-Type: application/json`. TLS enforcement is not configured in the library and must be enforced by the consuming service's URL configuration and infrastructure (e.g., Azure App Gateway, APIM with TLS termination).
- **Database connections:** R2DBC connection factory accepts URL-encoded properties. TLS for SQL Server connections must be specified in the `r2dbc.url` by the consuming service (e.g., `encrypt=true;trustServerCertificate=false`). The library does not enforce TLS at the connection factory level — this is a gap.

## Retention Concerns

As a library, this module does not generate or retain data. Log data generated by consuming services using this framework's structured logging configuration (Logstash JSON) will be retained according to each consuming service's operational policy. Log retention for PCI DSS Requirement 10.7 mandates at least 12 months (3 months immediately available) — this must be configured at the log aggregation infrastructure level.

## PCI DSS Compliance Assessment

- Req 3 (Stored Data): No CHD stored. Dapr secret abstraction aligns with credential protection.
- Req 6.3 (SBOM): CycloneDX plugin produces an aggregate BOM — strong alignment.
- Req 8.6 (Secrets): Dapr secret store pattern aligns with non-hardcoded credentials requirement.
- Gap: R2DBC TLS enforcement is not mandated by the library; individual services must configure `encrypt=true` explicitly.
- Gap: Secrets `MutableMap` in JVM heap is not zeroed post-use, which is a minor memory hygiene risk in a PCI CDE context.
