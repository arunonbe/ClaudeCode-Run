# Enterprise Architect View — request-file_LIB

## Platform Generation

**Gen-1 (eCount/Citi)** — confirmed by:
- `groupId: com.ecount.service` — eCount namespace
- `parent: com.parents:prepaid-parent:6.0.12` — Gen-1 eCount parent BOM
- XML-file-based batch processing pattern (pre-REST era architecture)
- JAXB classes in `com.ecount.payment` namespace with `http://www.ecount.com/` XML namespace
- Spring XML application context wiring (`applicationContext-requestfile.xml`)
- Dependency on legacy artifacts (`commons-lang`, `sax`, `com.sun.xml.bind:jaxb-xjc`)
- `XStream` library usage (common in early-2000s Java XML processing)

## Integration Patterns

The library implements the **File-Based Messaging** integration pattern (EIP: File Poller / Document Message). This is the classical batch processing pattern for large-scale payment operations in the Gen-1 era:

1. The Onbe platform creates a batch of payment instructions as an in-memory object tree.
2. The library serializes the object tree to an XML file on a shared filesystem.
3. The payment processor (eCount/Citi backend) polls the shared filesystem directory for new XML files.
4. The processor reads, validates, and executes the payment instructions.
5. Result files (if any) are written back to another shared directory.

This pattern has no Gen-3 equivalent. Gen-3 services use event-driven messaging (Azure Service Bus), REST APIs, and real-time processing rather than file-based batch polling.

## External Dependencies

- **Shared filesystem** (`REQUEST_FILE_BASE_PATH`): The library writes files to a path configured at runtime. This path likely points to a shared network filesystem (NFS, CIFS/SMB) accessible to both the Onbe application server and the eCount payment processor. This shared filesystem is a critical integration point with no modern equivalent.
- **eCount payment processor**: The consumer of the XML files generated by this library. The processor is the Gen-1 Citi/eCount backend system.
- **Spring XML application context**: The library is wired via Spring XML beans; consuming services must configure `applicationContext-requestfile.xml`.

## Position in the Broader Platform

`request-file_LIB` is a **batch disbursement enablement library** in the Gen-1 payment processing pipeline. Its position:

```
[Sweep/Order Processing Service] 
    → assembles PaymentData (AccountCreationVO, FundsAdditionVO, etc.)
    → calls RequestBuilder.createReqFile()
    → [Shared Filesystem] (XML files written)
        → [eCount payment processor] reads and processes
            → [Fiserv / card network] card loads, emboss requests
```

The `FileOrderManagerSvcSpec` in `qa-test-automation` confirms that a `FileOrderManager` service uses file-based order submission — `request-file_LIB` is the underlying library for that service.

## Migration Blockers

The file-based integration pattern is the primary migration blocker. It requires:
1. **Shared filesystem dependency**: Modern containerized (Azure Container Apps, AKS) deployment models cannot reliably use shared NFS/SMB filesystems without Azure NetApp Files or Azure Files integration — adding infrastructure complexity.
2. **Processor coupling**: The eCount payment processor must consume the specific XML schema produced by this library. Changing the format requires simultaneous updates on both sides.
3. **Batch-vs-realtime**: The file-based pattern is inherently batch-oriented. Migrating to Gen-3 requires re-architecting to event-driven (Azure Service Bus messages) or REST-based payment instruction submission, which changes the timing model for all downstream operations.
4. **PCI DSS compliance gap**: The CVV marshalling issue (see Solution Architect analysis) must be resolved before any migration or continued use. This may require schema changes that affect the processor side.

## Strategic Status

**Legacy, high compliance risk, requires immediate action on CVV finding.** This library:
1. Implements a filing pattern that is architecturally incompatible with Gen-3 containerized deployment.
2. Contains a potentially critical PCI DSS Req 3.2.1 violation (CVV in XML files).
3. Depends on XStream (CVE-prone), legacy JAXB Sun artifacts, and Commons Lang 1.x.

Recommended actions:
1. **Immediate**: Audit all callers of `Cardtype.setCvcode()` across the Gen-1 codebase to determine if CVV is ever populated. If populated, this is a live PCI DSS violation requiring immediate remediation.
2. **Short-term**: Add `@XmlTransient` to `Cardtype.cvcode` to prevent XML marshalling regardless of field population.
3. **Medium-term**: Migrate to an event-driven payment instruction pattern compatible with Gen-3 infrastructure.
4. **Long-term**: Sunset this library when the last Gen-1 batch processing service is decommissioned.
