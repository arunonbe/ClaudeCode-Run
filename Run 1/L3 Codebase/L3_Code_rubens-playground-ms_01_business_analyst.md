# Business Analyst Analysis: rubens-playground-ms

## Business Purpose
rubens-playground-ms is a Gen-3 developer playground microservice used to prototype and demonstrate the StableRails Order API pattern within the Onbe / Nexpay platform. It implements a simplified prepaid card order creation workflow: an order is created with cardholder account details, persisted to a database, and a reward notification email is sent to the recipient.

The name "StableRails" and the parent POM `nexpay-parent` indicate this is part of the Nexpay Gen-3 platform development track. The "-playground" designation and "rubens" prefix (a developer's name) confirm this is a sandbox/prototype, not a production system.

## Capabilities
1. **Create Order** (`POST /orders`): Accept an order request with cardholder name, address, contact, and payment details; persist to the `orders` table; return an order ID, smart link GUID, and line item ID.
2. **Get Order by ID** (`GET /orders/{orderId}`): Retrieve a persisted order by its formatted orderId (pattern `ORD_\d{6}`).
3. **Lookup Order** (`GET /orders`): Look up a single order by either orderId or smartLinkGuid (exactly one required).
4. **Send Reward Email**: On successful order creation, render a FreeMarker HTML email template and send via SendGrid.
5. **Database Schema Management**: Liquibase manages the `orders` table DDL and sequence generation.
6. **Health Check**: Spring Boot Actuator `/actuator/health` endpoint.
7. **API Documentation**: Springdoc OpenAPI (`/swagger-ui`) served from the bundled `ordersapi.yml`.

## Key Business Entities
| Entity | Description |
|---|---|
| Order | Core entity: all order fields stored in `orders` table |
| orderId | DB-computed formatted identifier: `ORD_000001` pattern |
| smartLinkGuid | DB-generated UUID: shareable URL for reward redemption |
| lineItemId | DB-computed formatted identifier: `LIN_000001` pattern |
| sequenceNumber | PK BIGINT from `orders_number_sequence` |
| status | Order lifecycle state: Created / Processing / Completed / Failed |
| program | Program name associated with the order |
| promotion | Optional promotion code |
| accessLevel | Card access level (e.g., "Plastic") |
| currency | ISO 4217 code (e.g., "USD") |
| firstName / lastName | Cardholder name |
| dateOfBirth | Cardholder date of birth |
| address1/2, city, state, postalCode, country | Cardholder address |
| primaryPhone / mobilePhone / email | Cardholder contact |
| amount | Payment amount (DECIMAL 18,2) |
| addPayment | Whether initial load is requested |
| memo | Free-form order note |

## Business Rules
- `orderId` follows pattern `ORD_\d{6}` (API validation on GET); generated by the database.
- `smartLinkGuid` is a UUID generated by the database using `RANDOM_UUID()`.
- Exactly one of `orderId` or `smartLinkGuid` must be provided for the lookup endpoint; providing both or neither returns HTTP 400.
- Email sending failures do not fail the order creation — they are logged and swallowed.
- SendGrid can be disabled via configuration (`SENDGRID_ENABLED=false`); orders still succeed with email skipped.
- `program`, `orderName`, and `currency` are required fields in the order request.
- Address requires `address1`, `city`, `country`, and `postalCode`.
- Contact requires `primaryPhone`.
- Smart link base URL is configurable via `app.smartlink.baseUrl` and must be an HTTP(S) URL (validated by `@Pattern`).

## Business Flows
### Order Creation Flow
1. Client sends `POST /orders` with `CreateOrderRequest` body.
2. `OrdersApiDelegateImpl.createOrder()` maps request to `Order` entity via `OrderMapper`.
3. `OrderRepository.saveAndFlush()` persists order; database computes `orderId`, `smartLinkGuid`, `lineItemId`, `createdDate`.
4. If email is configured and enabled, `sendRewardEmail()` renders FreeMarker template and sends via SendGrid.
5. Returns `CreateOrderResponse` with order details and HTTP 201.

### Email Notification Flow
1. `buildEmailTemplateVariables()` creates template context: company name (program), recipient name, amount, smart link accept URL, support email, year.
2. FreeMarker renders `reward-email.ftlh` template to HTML.
3. `SendGridEmailServiceImpl.sendHtmlEmail()` sends via SendGrid Java SDK.

## Compliance Relevance
- Collects and persists PII: name, address, phone, email, date of birth.
- `dateOfBirth` persisted in the `orders` table — CCPA/GDPR sensitive data element.
- SendGrid transmits PII (name, email) to a third-party email provider — data sharing agreement required.
- No PAN collection or storage visible in this service.
- Playground status means this service should NOT handle production cardholder data.

## Risks (Business Perspective)
- **Playground in production**: If this service were promoted to production without security hardening, PII collected here would not have appropriate controls.
- **No authentication**: API is currently unauthenticated (bearer auth defined in OpenAPI spec but enforcement not visible in code).
- **Date of birth persistence**: DOB stored in the orders table without encryption.
- **SendGrid API key**: Must be configured securely; exposure would allow sending emails on behalf of Onbe.
