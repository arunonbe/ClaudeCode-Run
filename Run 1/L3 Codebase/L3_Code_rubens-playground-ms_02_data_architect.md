# Data Architect Analysis: rubens-playground-ms

## Data Stores
| Store | Type | Purpose |
|---|---|---|
| PostgreSQL (`ordersvc` database — DS_DB_ordersvc) | Relational DB | Primary data store for all order records |
| H2 (in-memory) | Embedded DB | Local development and test execution only |
| SendGrid | External email service | Transactional email delivery; transmits PII (name, email) |
| Azure Container Apps | Compute platform | Implied by `app.smartlink.baseUrl` default value pointing to Azure Container Apps domain |

## Sensitive Data Handled
| Data Element | Classification | Notes |
|---|---|---|
| first_name / last_name | PII | Persisted to DB; transmitted to SendGrid |
| email | PII | Persisted; transmitted to SendGrid |
| date_of_birth | PII (sensitive) | Persisted in DB in plain DATE column |
| address1/2, city, state, postal_code, country | PII | Persisted to DB |
| primary_phone / mobile_phone | PII | Persisted to DB |
| unique_id | External reference | May be PII depending on format |
| amount | Financial | Payment amount |

No PANs, CVVs, SSNs, or account numbers are collected or stored by this service.

## Encryption
### At Rest
- No application-level column encryption visible in the `Order` entity.
- `date_of_birth` is stored in a plain `DATE` column without encryption.
- Database-level encryption (PostgreSQL TDE or column encryption) is not configured within this service; depends on the PostgreSQL deployment.
- Liquibase-managed schema does not include column encryption DDL (would need inspection of `db/` changelog files to confirm).

### In Transit
- HTTPS enforced via Spring Boot configuration (not visible in this repo; depends on deployment).
- SendGrid calls via SendGrid Java SDK (`sendgrid-java:4.10.3`) — SDK uses HTTPS.
- PostgreSQL connection encryption depends on JDBC connection string (not visible in code; set via environment variable `DB_HOST`, `DB_USER`, `DB_PASSWORD`).

## Data Flow
```
Client (REST)
  --> POST /orders (JSON)
  --> OrdersApiDelegateImpl.createOrder()
  --> OrderMapper.toEntity(CreateOrderRequest) → Order
  --> OrderRepository.saveAndFlush(order) → PostgreSQL (orders table)
  <-- Order (with DB-generated orderId, smartLinkGuid, lineItemId, createdDate)
  --> sendRewardEmail(order)
        --> TemplateService.renderTemplate("reward-email.ftlh", vars)
        --> SendGridEmailServiceImpl.sendHtmlEmail(EmailRequest)
              --> SendGrid API (HTTPS) → Recipient mailbox
  <-- CreateOrderResponse → 201 Created
```

## Data Quality / Retention
- `orderId` and `smartLinkGuid` are database-generated and unique per row; natural deduplication at DB level.
- No soft-delete or retention policy visible; records persist indefinitely.
- `DataIntegrityViolationException` is caught at `OrdersApiDelegateImpl.java:139` and re-thrown (not swallowed) — DB constraint violations surface as errors.
- No data validation beyond bean validation (`@NotBlank`, `@Pattern`) on the API layer.

## Compliance Gaps
1. **Date of birth in plain column**: `date_of_birth` persisted without encryption; CCPA/GDPR categorises DOB as sensitive personal data.
2. **PII in `memo` field**: Free-form 4000-character memo field may contain arbitrary PII submitted by the caller; no content scanning or sanitisation.
3. **SendGrid PII transmission**: Name and email transmitted to SendGrid (third-party); must be covered by DPA and privacy notice.
4. **No data retention / deletion**: No TTL or GDPR right-to-erasure mechanism visible.
5. **DB credentials via environment variables**: `DB_PASSWORD` passed as env var in `docker-compose.yml`; must be managed via secrets vault in production.
6. **Playground data in shared DB**: If the `ordersvc` database is shared with other services, test/playground data pollutes the namespace.
