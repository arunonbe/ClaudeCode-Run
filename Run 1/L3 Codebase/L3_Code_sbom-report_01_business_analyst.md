# Business Analyst View — sbom-report

## Business Purpose

`sbom-report` is an enterprise-wide Software Bill of Materials (SBOM) aggregation and reporting repository for Onbe's technology platform. It collects per-repo CycloneDX SBOMs generated by individual service CI/CD pipelines, aggregates them into a cross-platform dependency inventory, and produces consolidated reports that enable Onbe's security and compliance teams to identify third-party library exposure, license obligations, and vulnerable dependency versions across the multi-generation codebase.

SBOMs are a regulatory and security best-practice requirement. NIST SP 800-161 (supply chain risk management), Executive Order 14028 (improving the nation's cybersecurity), and PCI DSS 4.0 Req 6.3.3 (keep all software components protected from known vulnerabilities) all point toward SBOM-based dependency transparency as a control.

## Capabilities

- **SBOM collection**: Stores individual CycloneDX 1.4/1.7-format SBOM JSON files for each Onbe service component, organized by technology stack (currently `dotnet/` for .NET services; Java and JavaScript directories are processed by the aggregation script based on `js` and `java` arguments).
- **Cross-platform aggregation**: The `report_aggregated.py` Python script reads all SBOM JSON files in a given directory, extracts component metadata (name, group, version, publisher, description, licenses), deduplicates entries, and produces a consolidated CycloneDX BOM JSON file.
- **Onbe-internal filtering**: The aggregation script explicitly excludes Onbe-internal components (group IDs starting with `com.citi`, `com.ecount`, `com.onbe`, `com.web.api.accountmanagementapi-payout`) to focus the aggregated report on third-party dependencies only.
- **License tracking**: Extracts license identifiers (SPDX format) from individual SBOMs into the aggregated report, supporting OSS license compliance review.
- **Scheduled automated generation**: The GitHub Actions workflow runs weekly (Tuesday at 19:47 UTC) to regenerate aggregated reports and publish them as GitHub Actions artifacts.

## Client and Cardholder Impact

This repository has no direct client or cardholder impact. Its business value is internal risk management — ensuring Onbe's engineering and security teams have visibility into the third-party software components embedded in production services. Indirect cardholder impact occurs when the SBOM data identifies vulnerable dependencies (e.g., Log4Shell, Spring4Shell) that could compromise production systems handling cardholder data.

## Business Rules in Code

The key business logic in `report_aggregated.py`:
- Components with group IDs starting with Onbe-internal prefixes are excluded (filtering logic at line 34)
- Deduplication is performed by `bom-ref` to avoid double-counting identical dependency versions
- Output is sorted by `bom-ref` for consistent, diffable reports
- The aggregated output format is CycloneDX 1.4 (regardless of input SBOM version, which may be 1.7 for .NET components)

## Regulatory Obligations

- **PCI DSS 4.0 Req 6.3.3**: All system components must be protected from known vulnerabilities by installing applicable security patches. SBOM reports are the mechanism for identifying which components have unpatched CVEs.
- **NIST CSF 2.0 / NIST SP 800-161**: Software supply chain risk management requires inventory of all software components.
- **Executive Order 14028 / CISA SBOM guidance**: While not directly applicable to Onbe (not a federal contractor), the standards represent best practice and may become contractual obligations with government-adjacent clients.
- **Open Source License Compliance**: License data in the SBOMs (GPL, LGPL, MIT, Apache 2.0, etc.) is tracked to ensure Onbe's distribution obligations are met — copyleft licenses in service dependencies may create disclosure or code-release obligations.

## Key Business Risks

1. **Weekly batch cadence**: The aggregated report is generated only weekly. A critical CVE (e.g., Log4Shell) disclosed mid-week may not be captured in the aggregated report until the next scheduled run. For high-severity vulnerabilities, the SBOM pipeline should support on-demand or push-triggered generation.
2. **Incomplete SBOM coverage**: The repository contains only .NET SBOMs in the visible file set. Java and JavaScript SBOMs are referenced in the aggregation script (`python3 ./report_aggregated.py js`, `java`) but the corresponding directories are not present in the checked-out working tree. This may indicate incomplete SBOM collection across the 363-repo Onbe corpus.
3. **No CVE correlation**: The aggregation script merges component metadata but does not cross-reference against vulnerability databases (NVD, OSV). The aggregated SBOM identifies what is deployed but does not identify which components are vulnerable. A separate CVE matching step (e.g., OWASP Dependency-Check, Grype, Snyk) is required for actionable vulnerability management.
4. **Artifact retention**: Aggregated reports are published as GitHub Actions artifacts, which have a default 90-day retention limit. Long-term SBOM retention for audit purposes requires publishing to a persistent store (Azure Blob, GitHub Releases, a dedicated artifact registry).
