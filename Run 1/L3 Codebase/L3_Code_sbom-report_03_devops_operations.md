# DevOps / Operations View — sbom-report

## Build System

No build system in the traditional sense. The repository contains:
- A Python 3 script (`report_aggregated.py`) — no dependencies beyond the Python standard library (`os`, `json`, `uuid`, `datetime`, `argparse`)
- CycloneDX SBOM JSON files committed directly to the repository (input data)
- A GitHub Actions workflow that runs the Python script

No `requirements.txt`, `pyproject.toml`, `setup.py`, or other Python dependency manifest is present — the script uses only standard library modules, making dependency management unnecessary for the script itself.

## CI/CD Pipeline

GitHub Actions workflow defined at `.github/workflows/aggregated_report.yml`:

**Triggers**:
- Manual dispatch (`workflow_dispatch`)
- Scheduled: Tuesdays at 19:47 UTC (`cron: '47 19 * * 2'`)

**Steps**:
1. `actions/checkout@v4` — checks out the repository
2. Run Python aggregation script for `js`, `java`, and `dotnet` directories:
   ```bash
   mkdir -p ./aggregated-reports
   python3 ./report_aggregated.py js
   python3 ./report_aggregated.py java
   python3 ./report_aggregated.py dotnet
   ```
3. `actions/upload-artifact@v4` — uploads `aggregated-reports/*.json` as artifact named `aggregated-reports`

**GitHub Actions permissions** (defined in workflow):
- `actions: read`
- `contents: read`
- `security-events: write` — the `security-events: write` permission is present but no `security-events` upload step (e.g., `github/codeql-action/upload-sarif`) is visible in the workflow. This permission may be a holdover from a template or may be intended for future use.

**Timeout**: `${{ (matrix.language == 'swift' && 120) || 360 }}` — this timeout configuration references a `matrix.language` variable that is not defined in this workflow (there is no matrix strategy). This appears to be a copy-paste from a CodeQL workflow template. The effective timeout for this workflow is 360 minutes (6 hours), which is far longer than necessary for a Python script processing JSON files. This should be reduced to 15–30 minutes.

## Deployment Model

The aggregated reports are not deployed as a service. They are:
1. Generated by the GitHub Actions runner
2. Uploaded as GitHub Actions artifacts
3. Downloaded on-demand by security/compliance team members

There is no persistent hosting, API, or integration with a dedicated SBOM management platform (e.g., OWASP Dependency-Track, Anchore Enterprise, Snyk).

## Runtime Details

- **Runtime**: Python 3 (version determined by `ubuntu-latest` GitHub Actions runner — currently Python 3.12.x on ubuntu-latest)
- **No JVM, no Docker, no Spring Boot**
- **Script execution time**: Seconds to minutes depending on the number of SBOM files in each directory
- **Memory**: Minimal — JSON files are loaded one at a time; the `aggregated_data` list accumulates in memory but SBOM files are typically small

## Secrets Management

No secrets are required for the aggregation workflow itself. GitHub Actions standard token (`${{ github.token }}` — implicitly available) provides the permissions declared in the workflow. No external credentials, API keys, or service accounts are used.

## Observability

- `print()` statements in `report_aggregated.py` report each processed file to stdout — captured in GitHub Actions logs
- No structured logging, no metrics, no alerting
- Pipeline success/failure is observable via GitHub Actions workflow status (email notifications if workflow fails)

**Missing**: No alerting or notification if the aggregation script fails to process a file (e.g., malformed JSON from a service's SBOM CI step). The Python script will raise an unhandled exception on `json.load()` failure, which will fail the workflow step, but only passive monitoring (email on failure) is available.

## EOL and CVE Concerns

- **Python 3**: Actively maintained; no EOL concern on `ubuntu-latest`.
- **Python standard library only**: No third-party Python dependencies means no Python dependency CVEs.
- **CycloneDX spec version mismatch**: Input SBOMs are CycloneDX 1.7 (.NET); output is CycloneDX 1.4. The downgrade is applied without explicit validation that the 1.7 fields used in inputs are all present in 1.4. If future .NET SBOMs use 1.7-exclusive fields, the aggregated output may silently drop data.
- **`ubuntu-latest` runner**: GitHub periodically updates `ubuntu-latest`, which could change the Python version or introduce behavior changes. Pinning to `ubuntu-24.04` would provide more deterministic behavior.
- **GitHub artifact retention (90 days)**: Not a CVE but a compliance risk — aggregated reports older than 90 days are automatically deleted. For PCI DSS annual audit requirements, longer retention is needed.
