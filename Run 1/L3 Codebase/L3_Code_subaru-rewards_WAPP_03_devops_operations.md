- Log4j 1.x is EOL and has known critical CVEs (Log4Shell does not apply to Log4j 1.x but Chainsaw / Socket appender vulnerabilities do).
- No metrics, health check endpoint, or distributed tracing observed.
- State machine logs state transitions at DEBUG level.
- Exception stack traces written via `e.printStackTrace()` (not via logger) in multiple places.

## Operational Risks
| Risk | Severity |
|------|----------|
| VBScript scheduler (`run_subaru_rewards.vbs`) — Windows-only; no cross-platform or container-native scheduling | High |
| Log4j 1.x (EOL) — no security patches | High |
| Spring 2.0.2 — EOL (2008); no security patches | Critical |
| Spring Modules (unmaintained since ~2009) used for caching integration | High |
| `e.printStackTrace()` instead of structured logging — stack traces may be lost in some log configurations | Medium |
| `testLogin` property wired into production bean — if set, it overrides sales person ID logic | High |
| Three modules commented out of parent POM build — unknown stability; dead code in repository | Medium |
| XML payment request files written to filesystem with no encryption — sensitive data at rest | High |
