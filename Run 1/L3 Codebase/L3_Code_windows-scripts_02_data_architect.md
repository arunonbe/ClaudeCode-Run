# Data Architect Report — windows-scripts

## Data Models

No formal data models are defined in the scripts. The scripts operate against the eCount Core data model via the J2COM bridge. Key data objects referenced:

- **SecureProfile**: `federal_id` (SSN), `date_of_birth` — accessed via `ECountCore.eMember.UpdateSecureProfile` / `InquirySecureProfile`.
- **Member**: `member.id` (UUID format) — the cardholder's primary identifier.
- **ACH batch**: origination and return file records validated against NACHA format.
- **FDR reports**: CD/DD/SD series report structures (account journal, embossing, non-monetary, hold reports).
- **Inventory**: Personix card inventory and return mail records.

## Sensitive Data Identified

| Data Type | Script/File | Handling |
|-----------|-------------|---------- |
| Social Security Number (SSN) | `1099_ssn_update.vbs`, `p-az-app12/.../1099_ssn_update.vbs` | Passed as plaintext command-line argument; updated via RPC to eCount Core |
| Date of Birth | `dob_update.vbs` | Passed as plaintext command-line argument |
| ACH account/routing data | BCP format files (`ach_process_batch_request.fmt`, `fdr_process_nacha_file.fmt`, etc.) | Processed in bulk SQL Server BCP operations |
| IVR Pre-check data | `IVRPreCheck_*.fmt`, `IVRClaimable_file.fmt` | BCP format for cardholder pre-check eligibility data |
| Cardholder address | `fdr_dcaf_chd_data.fmt` (DCAF = Daily Cardholder Address File) | BCP import |
| Card status | `fdr_dcaf_status.fmt`, `arroweye_status_table.fmt` | Card production status data |
| DDA account journal | `fdr_dda_account_journal.fmt` | Direct Deposit Account journal records |

## Encryption Status

**No encryption is applied to scripts or the data they process at rest.** Key findings:

- SSN values are passed as plaintext command-line arguments to VBScripts. On Windows, command-line arguments are visible in process lists and may be logged by system monitoring tools.
- PGP encryption scripts (`pgp.vbs`) exist and are used for file transfer — indicating that outbound files are PGP-encrypted before transmission to banks. However, the scripts themselves are not encrypted.
- BCP format files define the structure of SQL Server bulk copy operations. The data being loaded/exported is not encrypted in transit within the data center network.
- The `citiReportParser.pl` and `fdrReportParser.pl` scripts read financial settlement report files from the filesystem without encryption.

## Database Schemas

Referenced implicitly through BCP format files and stored procedure calls:
- **eCount Core database**: Accessed via `ECountService.DataEnvironment.Execute` — VBScript invokes stored procedures by name/database.
- **NACHA processing tables**: `citi_process_nacha_file.fmt`, `fdr_process_nacha_file.fmt` map to SQL Server tables for NACHA file staging.
- **ACH tables**: `ach_process_batch_request.fmt` maps to the ACH batch request staging table.
- **Card production tables**: ArrowEye and CPS format files map to card issuance and check production staging tables.
- **IVR tables**: `IVRPreCheck_*.fmt` files map to IVR pre-check eligibility staging tables.

## Data Flows

1. **SSN/DOB correction**: Operator → VBScript → J2COM bridge → XML-RPC → eCount Core → `ecountcore` database (secure_profile table).
2. **ACH origination**: eCount Core → NACHA file → BCP export → bank transmission.
3. **ACH returns**: Bank → NACHA return file → BCP import → `ach_process_batch_request` table → eCount Core processing.
4. **FDR report ingestion**: FDR → settlement report file → Perl parser → SQL Server → eCount Core reconciliation.
5. **Card fulfilment**: Personix → fulfilment report file → Perl parser → SQL Server → inventory/status update.

## Retention Concerns

- Scripts that process SSN, DOB, and ACH data constitute PII/financial processing artefacts. Log files generated by these scripts may contain PII that is subject to GLBA and CCPA retention and deletion requirements.
- BCP format files reference specific table structures in the production database. If these tables are migrated or renamed during Gen-3 migration, the BCP scripts will fail silently or incorrectly.

## PCI DSS Compliance

- **Req. 3.4**: SSNs and DOBs must be protected at rest. Passing SSN as a plaintext command-line argument violates the spirit of this requirement.
- **Req. 7**: Access to SSN-manipulation scripts must be restricted to authorized personnel only. No access control evidence is present.
- **Req. 10**: Audit trails for SSN/DOB changes are not enforced by the scripts themselves; reliance on OS-level audit logging is assumed but not confirmed.
