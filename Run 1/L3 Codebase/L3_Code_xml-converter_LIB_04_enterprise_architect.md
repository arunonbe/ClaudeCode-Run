# Enterprise Architect View — xml-converter_LIB

## Platform Generation
**Gen-1** — Legacy Windows desktop tooling. Predates any modern service-oriented or cloud architecture.

## Business Domain
Prepaid Card Operations / Client Onboarding and Funding. Supports operational workflows for creating and funding cardholder accounts on the eCount/North Lane platform.

## Role in the Platform
Operator-side data preparation tool. Bridges the gap between client-supplied flat data (spreadsheets, CSV) and the XML request format required by the eCount job service. Acts as the "on-ramp" for batch account creation and disbursement operations.

## Dependencies
| Dependency | Type | Notes |
|---|---|---|
| eCount Job Service | Downstream consumer | Receives the XML files generated by this tool |
| North Lane XML schema | Implicit schema | XML element ordering is hardcoded to comply with North Lane processor requirements |
| `.Northlane` template files | Runtime data | Created and managed by internal operations staff |
| `fields.txt` | Runtime data | Master field definitions; origin/maintenance process not evident in repo |
| Windows OS | Platform | WinForms; Windows-only |
| .NET Framework 2.0 | Runtime | EOL; no longer supported by Microsoft |

## Integration Patterns
- File-based integration: output is XML files written to disk, picked up externally for submission
- No API calls, no messaging, no service bus
- Legacy "template + data merge" pattern (predates ETL tools and modern data pipelines)

## Strategic Status
**Retire / Replace.** Key indicators:
- .NET Framework 2.0 is end-of-life
- Desktop WinForms application — cannot be containerised, scaled, or monitored
- No CI/CD, no automated tests, no audit trail
- Business function (batch XML generation for job service) could be replaced by a web-based or API-driven submission portal
- Wirecard branding remnants indicate pre-Onbe heritage

## Migration Blockers
- Operator dependency: internal staff and potentially clients are trained on this tool's template format and workflow
- `fields.txt` and `.Northlane` files represent undocumented schema knowledge; these must be migrated or re-expressed in a modern format
- The XML element ordering rules (`regSort`, `fundsSort`, etc.) are hardcoded — they encode business rules that must be preserved in any replacement
- Generated XML format is dictated by the North Lane job service schema — any replacement must produce identical output
